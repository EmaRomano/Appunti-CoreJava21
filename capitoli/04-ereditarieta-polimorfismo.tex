\chapter{Ereditarietà e Polimorfismo}

\section{Classi, Superclassi e Sottoclassi}

L'ereditarietà è il meccanismo che permette di creare nuove classi basandosi su classi esistenti. L'idea fondamentale è quella di riutilizzare i metodi e i campi di una classe esistente, aggiungendo o modificando funzionalità per rispondere a esigenze più specifiche.

Il criterio fondamentale per stabilire se l'ereditarietà sia appropriata è la relazione \textbf{"is-a"} (è-un). 
Ogni istanza della sottoclasse deve poter essere considerata un'istanza della superclasse. Ad esempio, poiché ogni \textit{Manager} è un \textit{Dipendente}, la classe \texttt{Manager} può ereditare da \texttt{Employee}. Richiamiamo innanzitutto la classe \texttt{Employee} che abbiamo visto nei capitoli precedenti:

\begin{lstlisting}[language=Java, caption={La superclasse Employee originale}]
import java.time.LocalDate;

public class Employee {
    private String name;
    private double salary;
    private LocalDate hireDay;

    public Employee(String name, double salary, int year, int month, int day) {
        this.name = name;
        this.salary = salary;
        this.hireDay = LocalDate.of(year, month, day);
    }

    public String getName() {
        return name;
    }

    public double getSalary() {
        return salary;
    }

    public LocalDate getHireDay() {
        return hireDay;
    }

    public void raiseSalary(double byPercent) {
        double raise = salary * byPercent / 100;
        salary += raise;
    }
}
\end{lstlisting}

In Java, l'ereditarietà si esprime tramite la parola chiave \texttt{extends}.

\begin{lstlisting}[language=Java, caption={Esempio di ereditarietà}]
public class Manager extends Employee {
    private double bonus; // Campo aggiuntivo

    public void setBonus(double bonus) {
        this.bonus = bonus;
    }
}
\end{lstlisting}

Nonostante il prefisso "super", una superclasse non è "superiore" in termini di capacità. Al contrario, la sottoclasse è solitamente più complessa:
\begin{itemize}
    \item La \textbf{Superclasse} contiene i metodi più generali.
    \item La \textbf{Sottoclasse} eredita tutto ciò che c'è nella superclasse e aggiunge campi o metodi specializzati (\textit{campi e metodi propri}).
\end{itemize}

\begin{caution}[Precisazione sull'esempio del libro]
    Nel libro di testo, l'esempio viene proposto nell'ipotesi (puramente esemplificativa) che un Employee non possa mai diventare Manager. In una situazione reale questo non è in genere realistico e la modellazione delle classi dovrebbe tenere conto di questa possibilità.
\end{caution}

\begin{observation}[Accesso di classe vs Accesso di istanza]
    In Java, il modificatore \texttt{private} garantisce la visibilità a livello di \textbf{classe} e non di singola istanza:
    \begin{itemize}
        \item Un'istanza di \texttt{Employee} può accedere ai membri privati di un'altra istanza della stessa classe.
        \item Una sottoclasse \texttt{Manager}, pur ereditando lo stato, è considerata \textbf{codice esterno} rispetto alla definizione della superclasse.
    \end{itemize}
    \textbf{Conseguenza}: La sottoclasse possiede i dati della superclasse (memoria), ma deve interagire con essi tramite l'interfaccia pubblica (metodi), preservando l'integrità dell'incapsulamento.
\end{observation}

\section{Sovrascrivere i Metodi (Overriding)}

Spesso i metodi della superclasse non sono del tutto appropriati per la sottoclasse. Nel nostro esempio, il metodo \texttt{getSalary()} di \texttt{Employee} restituisce solo lo stipendio base, mentre per un \texttt{Manager} dovrebbe restituire la somma di stipendio e bonus.
Tentare di implementare il metodo accedendo direttamente al campo della superclasse fallisce:
\begin{lstlisting}[language=Java]
public double getSalary() {
    return salary + bonus; // ERRORE: salary e privato in Employee
}
\end{lstlisting}
Anche se \texttt{Manager} è una sottoclasse, deve rispettare l'incapsulamento e usare l'interfaccia pubblica della superclasse.
Un errore altro comune è chiamare il metodo per nome sperando che Java "capisca" di voler usare quello del padre:
\begin{lstlisting}[language=Java]
public double getSalary() {
    double baseSalary = getSalary(); // ERRORE: chiama se stesso all'infinito
    return baseSalary + bonus;
}
\end{lstlisting}
Questo codice causa un \textit{crash} del programma perché il metodo richiama se stesso senza sosta (Stack Overflow).

Per istruire il compilatore a invocare il metodo della superclasse anziché quello corrente, si utilizza la parola chiave \textbf{\texttt{super}}.

\begin{lstlisting}[language=Java, caption={Overriding corretto con super}]
public double getSalary() {
    double baseSalary = super.getSalary();
    return baseSalary + bonus;
}
\end{lstlisting}

\begin{important}[\texttt{super} non è un riferimento]
A differenza di \texttt{this}, che è un riferimento all'oggetto corrente (e può essere assegnato a una variabile), \textbf{\texttt{super}} è solo una \textbf{parola chiave speciale} che dirige il compilatore verso la superclasse. Non puoi, ad esempio, scrivere \texttt{Object x = super;}.
\end{important}

\begin{observation}[La Regola della Visibilità nell'Overriding]
Questa è una regola ferrea: \textbf{una sottoclasse non può rendere "meno visibile" un metodo ereditato.}:
Se un metodo nella superclasse è \texttt{public}, nella sottoclasse \textbf{deve} rimanere \texttt{public}. Non puoi trasformare un metodo pubblico del padre in un metodo privato o protetto nel figlio.
\end{observation}

\subsection{L'importanza di \texttt{@Override}}
Le annotazioni in Java sono metadati che forniscono informazioni aggiuntive al compilatore; saranno trattate più approfonditamente più avanti, ma è importante introdurre subito l'annotazione \texttt{@Override} in questo contesto.
Sebbene non sia strettamente necessaria per il funzionamento del codice, è buona norma precedere ogni metodo sovrascritto con l'annotazione \textbf{\texttt{@Override}}.

\begin{lstlisting}[language=Java, caption={Uso di @Override}]
@Override
public double getSalary() {
    return super.getSalary() + bonus;
}
\end{lstlisting}

[Motivazioni e Vantaggi]
L'uso di questa annotazione serve a istruire il compilatore sulle nostre intenzioni. Le motivazioni principali sono:
\begin{enumerate}
    \item \textbf{Controllo degli errori di battitura}: Se scrivessimo per errore \texttt{getsalary()} (con la 's' minuscola), senza annotazione Java penserebbe semplicemente che stiamo creando un \textit{nuovo} metodo. Con \texttt{@Override}, il compilatore segnalerebbe un errore perché non esiste alcun metodo \texttt{getsalary} nella superclasse da sovrascrivere.
    \item \textbf{Verifica della firma}: Se cambiamo i parametri del metodo (es. aggiungiamo un argomento), non stiamo più facendo \textit{overriding} ma \textit{overloading}. L'annotazione ci avvisa immediatamente se stiamo rompendo il legame con il metodo del padre.
    \item \textbf{Documentazione}: Rende immediatamente chiaro a chi legge il codice che quel metodo non è un'aggiunta originale della sottoclasse, ma una modifica di un comportamento ereditato.
\end{enumerate}



\begin{tip}[Fail-Fast]
Usare \texttt{@Override} è un esempio del principio \textit{"fail-fast"}: è meglio che il compilatore ci urli contro subito per un errore di distrazione, piuttosto che scoprire a runtime che il nostro \texttt{Manager} sta prendendo lo stipendio base perché il metodo di calcolo del bonus ha un nome leggermente sbagliato.
\end{tip}

\section{I Costruttori delle Sottoclassi}

A differenza dei metodi, i costruttori non vengono "ereditati" nel senso stretto del termine: ogni sottoclasse deve definire i propri costruttori, ma ha l'obbligo di invocare un costruttore della superclasse per inizializzare i campi privati ereditati. Il meccanismo di \textit{constructor chaining} (catena dei costruttori) impone infatti dei vincoli precisi a seconda della struttura della superclasse. Si possono verificare due scenari principali:

\begin{enumerate}
    \item \textbf{Superclasse con costruttore no-args (esplicito o implicito)}
    Se nella superclasse esiste un costruttore senza argomenti, la sottoclasse non è obbligata a dichiarare un costruttore proprio. In questo caso, viene utilizzato il costruttore di default della sottoclasse che invoca implicitamente \texttt{super()}.

    \item \textbf{Superclasse senza costruttore no-args}
    Se nella superclasse è presente almeno un costruttore con parametri e \textbf{non} è stato definito un costruttore senza argomenti, la sottoclasse ha un obbligo formale:
    bisogna dichiarare esplicitamente un costruttore nella sottoclasse e la prima istruzione di tale costruttore deve essere una chiamata esplicita a uno dei costruttori della superclasse tramite la sintassi \texttt{super(...parametri...)}.
\end{enumerate}

\begin{caution}[Errore di compilazione comune]
    Se la superclasse richiede parametri e la sottoclasse non li fornisce esplicitamente (o tenta di farlo dopo la prima riga), il compilatore segnalerà l'errore: \textit{"Implicit super constructor is undefined. Must explicitly invoke another constructor"}.
\end{caution}

\begin{observation}[Invocazione Implicita e Campi Propri]
    Se nella superclasse non è definito alcun costruttore esplicito (ovvero è presente il costruttore \textit{no-args} di default), la sottoclasse gode di maggiore flessibilità: è possibile definire un costruttore che inizializza esclusivamente i \textbf{campi propri} della sottoclasse. In questo scenario, la chiamata a \texttt{super()} può essere omessa (e di solito è considerata una buona pratica, per ridurre la verbosità), poiché il compilatore la inserirà automaticamente come prima riga invisibile.
\end{observation}


\begin{observation}[La regola della prima posizione]
    Entrambe le istruzioni, \texttt{this()} e \texttt{super()}, hanno il vincolo categorico di dover essere la \textbf{prima istruzione} del costruttore. Di conseguenza, \textbf{non possono mai comparire nello stesso costruttore}.
\end{observation}


\begin{observation}
Cosa succede se un costruttore nella sottoclasse usa \texttt{this(...)}? La responsabilità di invocare il padre viene semplicemente "passata" lungo la catena:
\begin{enumerate}
    \item Il Costruttore A chiama \texttt{this(...)} (Costruttore B).
    \item Il Costruttore B deve, a sua volta, chiamare \texttt{super(...)} (o un altro costruttore della stessa classe che alla fine lo faccia).
\end{enumerate}

Indipendentemente da quanti \texttt{this()} vengano usati, l'ultimo anello della catena di costruttori di una classe \textbf{deve} sempre invocare (esplicitamente o implicitamente) un costruttore della superclasse. La "nascita" del padre non può essere evitata, può solo essere delegata a un altro costruttore "fratello".
\end{observation}

\section{Polimorfismo}

Il polimorfismo è la capacità di un oggetto di essere trattato come un'istanza della sua superclasse. Questo concetto si basa sul \textbf{Principio di Sostituzione}.

Il principio afferma che è sempre possibile utilizzare un oggetto di una sottoclasse in qualsiasi contesto in cui sia attesa una superclasse. Questo perché, per la relazione \textit{is-a}, una sottoclasse possiede tutte le caratteristiche della superclasse.

\begin{lstlisting}[language=Java, caption={Esempio di sostituzione}]

// Sintassi generale:
// SuperClasse oggetto = new SottoClasse(...);

Employee e = new Manager("Rossi", 50000, 2023, 10, 1);
\end{lstlisting}

Nella riga di codice polimorfica di sopra, dobbiamo distinguere due aspetti:
\begin{itemize}
    \item \textbf{Tipo Dichiarato}: È il tipo della variabile (\texttt{Employee}). Determina quali metodi il compilatore ci permetterà di chiamare: \textit{non è possibile invocare metodi specifici} di \texttt{Manager} su una variabile dichiarata come \texttt{Employee}; se provassimo a chiamare \texttt{e.setBonus(10000)}, otterremmo un errore di compilazione.
    \item \textbf{Tipo Effettivo}: È il tipo dell'oggetto creato in memoria (\texttt{Manager}). Determina quale versione del metodo verrà eseguita a runtime.
\end{itemize}



\begin{important}[L'unidirezionalità della sostituzione]
    La sostituzione funziona solo verso l'alto nella gerarchia:
    \begin{itemize}
        \item \textbf{OK}: Assegnare un \texttt{Manager} a una variabile \texttt{Employee}.
        \item \textbf{ERRORE}: Assegnare un \texttt{Employee} a una variabile \texttt{Manager}.
    \end{itemize}
\end{important}

\begin{observation}[Array Polimorfici]
    Il principio di sostituzione si applica anche agli array. È possibile creare un array di una superclasse e riempirlo con oggetti di sottoclassi diverse:
    \begin{lstlisting}[language=Java]
    Employee[] staff = new Employee[3];
    staff[0] = new Manager("Boss", 80000, 2020, 1, 1);
    staff[1] = new Employee("Worker", 40000, 2021, 5, 1);
    \end{lstlisting}
    Questo permette di gestire gruppi eterogenei di oggetti con un unico ciclo di elaborazione.
\end{observation}

\subsection{Il Legame Dinamico (Dynamic Binding)}

Il cuore pulsante del polimorfismo è il \textbf{dynamic binding}. È il meccanismo per cui la JVM decide, solo al momento dell'esecuzione, quale versione di un metodo invocare.

Consideriamo un array di tipo \texttt{Employee} che contiene sia dipendenti "standard" che manager:

\begin{lstlisting}[language=Java, caption={Esempio di Dynamic Binding}]
Employee[] staff = new Employee[3];

// Sostituzione: Manager "travestito" da Employee
staff[0] = new Manager("Carl Cracker", 80000, 1987, 12, 15);
staff[1] = new Employee("Harry Hacker", 50000, 1989, 10, 1);
staff[2] = new Employee("Tony Tester", 40000, 1990, 3, 15);

// Ciclo polimorfico
for (Employee e : staff) {
    System.out.println(e.getName() + " " + e.getSalary());
}
\end{lstlisting}

Quando il programma arriva alla riga \texttt{e.getSalary()}, avviene la magia del dynamic binding:
\begin{enumerate}
    \item Il \textbf{Compilatore} controlla il tipo dichiarato (\texttt{Employee}) e verifica che esista un metodo \texttt{getSalary()}. Se esiste, dà il via libera.
    \item La \textbf{JVM (a runtime)} guarda l'oggetto reale a cui punta \texttt{e}:
    \begin{itemize}
        \item Se \texttt{e} punta a un \texttt{Employee}, viene eseguito \texttt{Employee.getSalary()} (quello base).
        \item Se \texttt{e} punta a un \texttt{Manager}, viene eseguito \texttt{Manager.getSalary()} (il metodo sovrascritto, che aggiunge il bonus).

    \end{itemize}
\end{enumerate}

\begin{observation}[Flessibilità ed Estensibilità]
    La bellezza del dynamic binding è che se domani aggiungessimo una nuova sottoclasse \texttt{Executive} con il suo metodo \texttt{getSalary()} specifico, il ciclo \texttt{for} qui sopra \textbf{non cambierebbe di una virgola}. Il codice diventa "aperto" a nuove forme senza dover essere riscritto.
\end{observation}

\begin{observation}
    Il polimorfismo è la capacità di una variabile di riferirsi a tipi diversi. Il dynamic binding è la capacità della JVM di scegliere il metodo corretto per quel tipo specifico. Sono due facce della stessa medaglia.
\end{observation}

\begin{observation}[Hiding dei Metodi Statici]
     \label{hiding_metodi_statici}
    Cosa succede se definiamo in una sottoclasse un metodo \texttt{static} con la stessa firma di un metodo \texttt{static} della superclasse? 
    
    A differenza dei metodi d'istanza, i metodi statici \textbf{non partecipano al polimorfismo}. In questo caso non si parla di \textit{overriding} ma di \textbf{\textit{hiding}} (nascondimento).

    \begin{itemize}
        \item \textbf{Binding Statico}: Il metodo da eseguire viene deciso dal compilatore in base al \textit{tipo dichiarato} della variabile, non al \textit{tipo effettivo} dell'oggetto.
        \item \textbf{Mancanza di Polimorfismo}: Se una variabile \texttt{Employee e} punta a un oggetto \texttt{Manager}, l'invocazione di un metodo statico tramite \texttt{e} chiamerà sempre la versione di \texttt{Employee}.
    \end{itemize}

    \begin{lstlisting}[language=Java, caption={Differenza tra Overriding e Hiding}]
    Employee e = new Manager();
    
    e.getSalary();      // Overriding: esegue il metodo di Manager (Runtime)
    e.staticMethod();   // Hiding: esegue il metodo di Employee (Compile-time)
    \end{lstlisting}

    \textbf{Best Practice}: Proprio perché il comportamento dei metodi statici può trarre in inganno quando usati con variabili polimorfiche, in Java è considerata cattiva pratica invocare metodi statici tramite un'istanza (es. \texttt{e.staticMethod()}). Si dovrebbe sempre usare il nome della classe (es. \texttt{Employee.staticMethod()}).
\end{observation}

\subsection{Sintesi Mnemonica: Il Senso di Marcia}
\label{subsec:sintesi}

Per evitare confusione nel gestire le gerarchie, possiamo definire una regola basata sulla "direzione" in cui Java percorre l'albero delle classi:

\begin{description}
    \item[\textbf{I Costruttori partono dall'alto (Top-Down)}:] 
    Quando crei un oggetto, Java garantisce che le fondamenta siano solide prima di costruire i piani superiori. L'esecuzione risale fino a \texttt{Object} e poi scende: \textit{Padre $\rightarrow$ Figlio}. Non puoi esistere come "Manager" se prima non sei stato inizializzato come "Employee".
    
    \item[\textbf{I Metodi partono dal basso (Bottom-Up)}:] 
    Quando invochi un metodo su un oggetto polimorfico, Java cerca l'implementazione più specifica possibile. Parte dalla classe effettiva dell'oggetto e risale solo se non trova una sovrascrittura: \textit{Figlio $\rightarrow$ Padre}. Si predilige sempre la "personalità" della sottoclasse rispetto alla generalità della superclasse.
\end{description}



\begin{tip}[Perché questo dualismo?]
    Questa differenza esiste per un motivo preciso:
    \begin{itemize}
        \item I \textbf{costruttori} servono alla sicurezza: garantiscono che ogni parte dell'oggetto sia configurata correttamente.
        \item I \textbf{metodi} servono alla flessibilità: permettono a una sottoclasse di cambiare comportamento senza che chi usa l'oggetto debba accorgersene.
    \end{itemize}
\end{tip}

\subsection{Covarianza dei Tipi di Ritorno}

Una regola particolare dell'overriding riguarda il tipo di ritorno del metodo. La firma del metodo sovrascritto deve in ogni caso rimanere identica, ed in teoria anche il tipo di ritorno dovrebbe essere lo stesso. Java però permette alla sottoclasse di restituire un \textbf{sottotipo} rispetto a quello dichiarato nella superclasse.

\begin{observation}[Definizione di Covarianza]
    Si parla di \textit{tipi di ritorno varianti} (o covarianza) quando una sottoclasse sovrascrive un metodo cambiando il tipo di ritorno con uno più specifico (una sua sottoclasse).
\end{observation}

\begin{example}[Esempio getBuddy]
Consideriamo i seguenti metodi:
\begin{itemize}
    \item Superclasse: \texttt{public Employee getBuddy() \{ ... \}}
    \item Sottoclasse: \texttt{public Manager getBuddy() \{ ... \}}
\end{itemize}
Questo codice è perfettamente legale perché \texttt{Manager} è un sottotipo di \texttt{Employee}. L'overriding è valido e il tipo di ritorno è considerato "compatibile".
\end{example}



\begin{important}[Vincolo di direzione]
    La covarianza funziona solo in una direzione: si può restituire qualcosa di \textbf{più specifico}, mai qualcosa di più generico. 
    \begin{itemize}
        \item Se il padre restituisce \texttt{Manager}, il figlio non può restituire \texttt{Employee}.
    \end{itemize}
\end{important}


\section{Prevenire l'Ereditarietà: Il modificatore \texttt{final}}

Il modificatore \texttt{final} può essere applicato a classi e metodi per impedirne, rispettivamente, l'estensione o la sovrascrittura.

\begin{lstlisting}[language=Java, caption={Sintassi di final}]
public final class Executive extends Manager { ... } // Classe non estensibile

public class Employee {
    public final String getName() { return name; } // Metodo non sovrascrivibile
}
\end{lstlisting}

\begin{important}[Distinzione tra Campi e Classi]
    Se una classe è \texttt{final}, i suoi \textbf{metodi} sono automaticamente \texttt{final}, ma i suoi \textbf{campi} NO. La mutabilità dei dati rimane regolata dai singoli modificatori sui campi.
\end{important}

Dunque l'effetto di dichiarare \texttt{final} classi o metodi è piuttosto semplice da capire; meno ovvio è capire quando e perché farlo. Esistono due motivazioni principali, una di \textbf{design} e una di \textbf{sicurezza}.

\subsubsection{1. Integrità della Semantica (Design)}
Si usa \texttt{final} quando il progettista vuole garantire che il comportamento di una classe non venga alterato o "corrotto" da sottoclassi.
\begin{itemize}
    \item \textbf{Esempio di \texttt{String}}: Se \texttt{String} non fosse \texttt{final}, qualcuno potrebbe creare una sottoclasse \texttt{EvilString} che sembra una stringa normale ma invia i dati a un server remoto. Raggiungere l'immutabilità e la sicurezza richiede che nessuno possa cambiare come funziona una stringa.
\end{itemize}

\subsubsection{2. Sicurezza nei Costruttori: Il problema della "Fuga di \texttt{this}"}
Chiamare un metodo sovrascrivibile all'interno di un costruttore è estremamente pericoloso a causa dello scontro tra la costruzione \textit{Top-Down} e il binding \textit{Bottom-Up} (vedi paragrafo \ref{subsec:sintesi}).

\begin{lstlisting}[language=Java, caption={Esempio del fallimento nel costruttore}]
public class Employee {
    public Employee(...) {
        // ... inizializzazione campi base ...
        System.out.println("Costruito: " + description());
    }

    public String description() {
        return "Dipendente con stipendio " + salary;
    }
}

public class Executive extends Employee {
    private String title; // Inizialmente null

    public Executive(String name, String title, ...) {
        super(name, ...); // Chiama il costruttore di Employee
        this.title = title;
    }

    @Override
    public String description() {
        // Se title e null, length() lancia NullPointerException
        return "Executive con titolo: " + title + " (lunghezza: " + title.length() + ")";
    }
}
\end{lstlisting}

\begin{description}
    \item[\textbf{Analisi del crash}:] 
    Quando esegui \texttt{new Executive(...)}:
    \begin{enumerate}
        \item Il costruttore di \texttt{Executive} invoca quello di \texttt{Employee} (\textbf{Top-Down}).
        \item Il costruttore di \texttt{Employee} chiama \texttt{description()}.
        \item Per il \textbf{Dynamic Binding}, la JVM cerca il metodo partendo dal basso e trova \texttt{Executive.description()}.
        \item Il metodo del figlio prova a leggere \texttt{title.length()}, ma il campo \texttt{title} non è ancora stato inizializzato (l'istruzione \texttt{this.title = title} verrà eseguita solo \textbf{dopo} che il costruttore del padre avrà finito).
        \item \textbf{Risultato}: Il programma crasha con un \texttt{NullPointerException}.
    \end{enumerate}
\end{description}



\begin{tip}[Soluzione]
    Per evitare questo comportamento imprevedibile su oggetti "parzialmente costruiti", i costruttori dovrebbero invocare solo metodi dichiarati come \textbf{\texttt{final}} (che non possono essere sovrascritti) o \textbf{\texttt{private}} (che sono implicitamente final).
\end{tip}

\begin{observation}[Metodi statici e Binding Statico]
    Sebbene vengano usati meno spesso per questo scopo, oltre ai metodi \texttt{private} e \texttt{final}, anche i \textbf{metodi statici} sono sicuri da invocare all'interno di un costruttore. 
    
    Questo accade per effetto del \textbf{\textit{hiding}} descritto nell'osservazione \ref{hiding_metodi_statici}: il metodo da eseguire viene determinato dal compilatore in base al tipo della classe e non dall'oggetto reale a runtime. Di conseguenza, non vi è il rischio di invocare accidentalmente logica della sottoclasse prima che questa sia stata inizializzata.
\end{observation}

\begin{tip}[Non abusare dei final!]
    In passato molti programmatori tendevano ad usare spesso \texttt{final} anche in contesti in cui non era necessario, per ragioni di ottimizzazione (che qui omettiamo). Oggi non è più necessario, perché la JVM moderna è già abbastanza intelligente da gestire l'ottimizzazione da sola. Dovresti usare \texttt{final} solo per ragioni di \textbf{chiarezza di design} e \textbf{sicurezza}.
\end{tip}


\section{Il Casting degli Oggetti}

Il casting consiste nel forzare il compilatore a trattare un riferimento di una superclasse come se fosse di una sottoclasse. Questa operazione è necessaria per accedere ai membri (campi o metodi) che sono presenti solo nella sottoclasse.

\begin{lstlisting}[language=Java, caption={Esempio di Downcasting}]
Employee e = new Manager(...); // Polimorfismo
// e.setBonus(5000); // ERRORE: Employee non ha setBonus

Manager m = (Manager) e; // Downcasting
m.setBonus(5000); // OK
\end{lstlisting}

Mentre l'\textit{upcasting} (da sottoclasse a superclasse) è sempre sicuro e automatico, il \textbf{downcasting} è una scommessa: se l'oggetto \texttt{e} fosse davvero un \texttt{Employee} "semplice" e non un \texttt{Manager}, l'istruzione \texttt{(Manager) e} causerebbe il crash immediato del programma con una \textbf{\texttt{ClassCastException}}.

Il casting non cambia l'oggetto in memoria. L'oggetto è sempre lo stesso. Quello che cambia sono gli "occhiali" con cui Java lo guarda:
\begin{itemize}
    \item Con gli occhiali \texttt{Employee}, vedi solo i metodi e campi dichiarati in \texttt{Employee}.
    \item Con gli occhiali \texttt{Manager}, vedi anche i metodi e campi dichiarati in \texttt{Manager}.
\end{itemize}

Il casting dice a Java: "Togli gli occhiali da Employee e metti quelli da Manager. Mi assumo io la responsabilità che dietro quel vetro ci sia effettivamente un Manager". Se ti sbagli, Java "ti punisce" interrompendo il programma.


\begin{tip}[Design Rule]
    Se ti ritrovi a voler chiamare un metodo specifico (es. \texttt{setBonus}) su una lista generica di oggetti (\texttt{Employee}), chiediti:
    \begin{enumerate}
        \item Quel metodo dovrebbe forse far parte della superclasse?
        \item Sto usando la struttura dati corretta o dovrei avere una lista separata di soli \texttt{Manager}?
    \end{enumerate}
    Il casting dovrebbe essere l'eccezione, non la regola: l'abuso del casting spesso è un segnale di un'architettura debole
\end{tip}

\section{Evoluzione di \texttt{instanceof}: Pattern Matching}
\label{instanceof}

A partire da Java 16, l'operatore \texttt{instanceof} è stato potenziato per ridurre la verbosità del \textit{downcasting} e migliorare la leggibilità del codice.

Invece di controllare il tipo e poi eseguire un cast manuale, è possibile dichiarare una \textbf{pattern variable} direttamente nel test.

\begin{lstlisting}[language=Java, caption={Confronto tra vecchio e nuovo approccio}]
// Prima di Java 16 (Verboso)
if (staff[i] instanceof Manager) {
    Manager boss = (Manager) staff[i]; // Cast manuale ripetitivo
    boss.setBonus(5000);
}

// Da Java 16 in poi (Pattern Matching)
if (staff[i] instanceof Manager boss) {
    boss.setBonus(5000); // boss e gia di tipo Manager
}
\end{lstlisting}

\subsection{Scoping e Operatori Logici}
La variabile introdotta dal pattern (es. \texttt{boss}) ha uno \textbf{scope limitato} basato sul flusso logico (\textit{flow-sensitive scoping}):

\begin{itemize}
    \item \textbf{AND (\texttt{\&\&})}: È permesso usare la variabile nella parte destra dell'espressione, poiché viene valutata solo se il test \texttt{instanceof} è vero.
    \item \textbf{OR (\texttt{||})}: È proibito, poiché la parte destra viene eseguita se il test è falso, rendendo la variabile non definita.
\end{itemize}

\begin{lstlisting}[language=Java]
if (e instanceof Executive exec && exec.getTitle().length() > 10) { ... } // OK
\end{lstlisting}

\begin{important}[Attenzione: Shadowing dei Campi]
    La variabile introdotta da un pattern (es. \texttt{exec}, \texttt{boss}) è una variabile locale. Come tale, può \textbf{oscurare} (shadow) un campo della classe che ha lo stesso nome. Questo può portare a errori logici difficili da individuare.

    \begin{lstlisting}[language=Java, caption={Esempio di Shadowing con instanceof}]
    class Value {
        private double v; // Campo della classe

        public boolean equals(Object other) {
            // Se il test e vero, 'v' all'interno del blocco
            // si riferisce a LabeledValue, NON al campo della classe.
            if (other instanceof LabeledValue v) {
                return this.v == v.getValue(); 
            }
            return false;
        }
    }
    \end{lstlisting}
    \textbf{Best Practice}: Per evitare confusione, è sempre consigliabile dare alla variabile del pattern un nome univoco o diverso dai campi della classe (es. \texttt{otherValue} invece di \texttt{v}).
\end{important}

\subsection{Il Record Pattern e la Decostruzione (Java 21+)}

L'evoluzione di Java ha cercato di rendere sempre più immediata la decostruzione dei records. 
Supponiamo, ad esempio, di avere il record: \texttt{record Point(double x, double y) \{\}}. Ecco come è cambiato il modo di estrarre le coordinate \texttt{x} e \texttt{y} da un oggetto \texttt{obj}.

\subsubsection*{1. Prima di Java 16}
Lavoro manuale in tre passaggi: controllo, cast manuale ed estrazione tramite metodi accessor.
\begin{lstlisting}[language=Java]
if (obj instanceof Point) {
    Point p = (Point) obj; // Cast fastidioso e ripetitivo
    double x = p.x();      // Estrazione manuale 1
    double y = p.y();      // Estrazione manuale 2
}
\end{lstlisting}

\subsubsection*{2. Java 16 - Type Pattern: Il primo passo}
Il cast scompare grazie alla \textit{pattern variable}, ma l'estrazione dei componenti è ancora manuale.
\begin{lstlisting}[language=Java]
if (obj instanceof Point p) {
    double x = p.x(); // Devo ancora chiamare esplicitamente i metodi
    double y = p.y();
}
\end{lstlisting}

\subsubsection*{3. Java 21 - Record Pattern: La Rivoluzione}
L'oggetto viene \textbf{decostruito} sul posto. Non serve più una variabile d'appoggio per l'intero oggetto se ci servono solo i suoi componenti.
\begin{lstlisting}[language=Java]
if (obj instanceof Point(double x, double y)) {
    // x e y sono variabili locali gia estratte e pronte all'uso!
    System.out.println("Coordinate: " + x + ", " + y);
}
\end{lstlisting}

\subsection*{Il "Superpotere": Il Nesting (Annidamento)}
Il vero vantaggio emerge con strutture dati complesse. Il Record Pattern permette di scendere in profondità in un'unica riga di codice.

\begin{lstlisting}[language=Java, caption={Decostruzione nidificata}]
record Point(double x, double y) {}
record Circle(Point center, double radius) {}

// Recupero la coordinata X del centro di un cerchio obj in un colpo solo:
if (obj instanceof Circle(Point(double x, double y), double r)) {
    System.out.println("La X del centro e: " + x);
}
\end{lstlisting}

\begin{observation}[Utilità nel Backend]
    Questa sintassi è estremamente potente quando si maneggiano strutture dati nidificate (come quelle provenienti da JSON o database). Permette di "interrogare" la forma dell'oggetto e contemporaneamente estrarre chirurgicamente solo i dati necessari, riducendo drasticamente il \textit{boilerplate code} e il rischio di errori.
\end{observation}

\begin{observation}[Variabili non usate]
    In Java 21, se un componente del record non è necessario, si può usare il carattere \textit{underscore} (\texttt{\_}) per ignorarlo (JEP 443):
    \texttt{if (p instanceof Point(var a, \_)) \{ ... \}}
\end{observation}

\section{Accesso Protetto: Il modificatore \texttt{protected}}

Il modificatore \texttt{protected} offre una via di mezzo tra il rigore del \texttt{private} e la totale apertura del \texttt{public}. Permette l'accesso ai membri di una classe alle sue sottoclassi e a tutte le classi all'interno dello stesso package.

Ecco un confronto dei quattro livelli di accesso in Java:

\begin{table}[h]
\centering
\begin{tabular}{|l|c|c|c|c|}
\hline
\textbf{Modificatore} & \textbf{Classe} & \textbf{Package} & \textbf{Sottoclasse} & \textbf{Mondo} \\ \hline
\texttt{public}       & Sì              & Sì               & Sì                   & Sì             \\ \hline
\texttt{protected}    & Sì              & Sì               & Sì* & No             \\ \hline
\textit{Default}      & Sì              & Sì               & No                   & No             \\ \hline
\texttt{private}      & Sì              & No               & No                   & No             \\ \hline
\end{tabular}
\caption{Tabella dei modificatori di accesso.}
\end{table}


Spieghiamo il significato dell'asterisco (*) in tabella. Esiste una regola sottile ma fondamentale: se una sottoclasse si trova in un \textbf{package diverso} dalla superclasse, essa può accedere ai membri \texttt{protected} \textbf{solo attraverso riferimenti del proprio tipo} (o dei propri sottotipi).

\begin{lstlisting}[language=Java, caption={Restrizione di accesso protected}]
// Package A
public class Employee {
    protected LocalDate hireDay;
}

// Package B
public class Administrator extends Employee {
    public void check(Employee e, Administrator a) {
        this.hireDay = ...; // OK: accesso al proprio campo ereditato
        a.hireDay = ...;    // OK: a e un Administrator
        // e.hireDay = ...; // ERRORE: non puo spiare dentro un Employee generico
    }
}
\end{lstlisting}

\begin{tip}[Considerazioni di Design: Campi vs Metodi]
\begin{itemize}
    \item \textbf{Campi \texttt{protected}}: Generalmente sconsigliati. Violano l'incapsulamento perché una volta dichiarati, non puoi più cambiare l'implementazione interna senza rompere il codice di tutte le sottoclassi scritte da altri.
    \item \textbf{Metodi \texttt{protected}}: Molto più utili. Indicano un metodo "delicato" che solo le sottoclassi (che conoscono bene il funzionamento interno del padre) sono autorizzate a invocare.
\end{itemize}
\end{tip}

\section{La Classe \texttt{Object}}

In Java, la classe \texttt{java.lang.Object} è l'antenata comune di tutte le classi. Se una classe non specifica esplicitamente una superclasse con \texttt{extends}, essa eredita automaticamente da \texttt{Object}. Dunque variabile di tipo \texttt{Object} funge da "contenitore universale" per qualsiasi riferimento a oggetto.

Ogni classe in Java eredita quindi tutti i metodi di questa "superclasse cosmica" \texttt{Object}. Tuttavia, poiché \texttt{Object} è un'astrazione totale, le sue implementazioni di default sono necessariamente generiche e basate sull'identità fisica dell'oggetto piuttosto che sul suo stato logico.

Affinché tali metodi possano eseguire azioni utili per le specifiche classi, è spesso cruciale effettuare l'override di questi metodi. Questa operazione è spesso delicata e bisogna sapere come eseguirla correttamente, altrimenti si rischia di introdurre bug difficili da diagnosticare.

Vedremo quindi come implementare correttamente i tre metodi principali: \texttt{equals()}, \texttt{hashCode()} e \texttt{toString()}.

\subsection{Il Metodo \texttt{equals()}}

Il metodo \texttt{equals()} della classe \texttt{Object} implementa di default il test di \textbf{identità}: due riferimenti sono uguali solo se puntano allo stesso identico indirizzo di memoria (\texttt{this == otherObject}).

Tuttavia, per molte classi è necessario un test basato sullo \textbf{stato}: due oggetti sono uguali se contengono gli stessi dati. Ad esempio, due stringhe con lo stesso testo dovrebbero essere considerate uguali anche se sono due oggetti distinti in memoria, oppure due array con gli stessi elementi dovrebbero essere considerati uguali.

Di seguito viene riportata l'implementazione del metodo \texttt{equals()} per \texttt{Employee}. Questo esempio rappresenta la struttura "tipo" che ogni sviluppatore dovrebbe seguire per garantire un confronto robusto.

\begin{lstlisting}[language=Java, caption={Implementazione standard di equals nella classe Employee}]
public class Employee
{
   /* ... campi della classe ... */
   @Override
   public boolean equals(Object otherObject)
   {
      // 1. Test rapido di identita: 
      // Se i riferimenti sono identici, gli oggetti sono certamente uguali.
      if (this == otherObject) return true;

      // 2. Controllo del valore null:
      // Per contratto, equals(null) deve sempre restituire false.
      if (otherObject == null) return false;

      // 3. Confronto delle classi:
      // Se le classi non coincidono esattamente, gli oggetti non sono uguali.
      // Si usa getClass() per garantire la simmetria nelle gerarchie.
      if (getClass() != otherObject.getClass())
         return false;

      // 4. Casting:
      // Arrivati a questo punto, sappiamo che otherObject e un 
      // Employee non nullo. Possiamo procedere al cast in sicurezza.
      Employee other = (Employee) otherObject;

      // 5. Confronto dei campi (Uguaglianza di stato):
      // Si confrontano i singoli campi significativi per determinare 
      // se il contenuto degli oggetti e lo stesso.
      return name.equals(other.name)
         && salary == other.salary
         && hireDay.equals(other.hireDay);
   }
}
\end{lstlisting}



\begin{observation}
    Da notare come il test di identità (\texttt{this == otherObject}) sia posto all'inizio come ottimizzazione: se stiamo confrontando un oggetto con se stesso, risparmiamo il tempo di eseguire tutti i controlli successivi e i confronti tra i campi.
\end{observation}

\begin{lstlisting}[language=Java, caption={Implementazione robusta di equals}]
public boolean equals(Object otherObject) {
    if (this == otherObject) return true; // 1
    if (otherObject == null) return false; // 2
    if (getClass() != otherObject.getClass()) return false; // 3
    
    Employee other = (Employee) otherObject; // 4
    
    // 5. Utilizzo di Objects.equals per gestire i campi null
    return Objects.equals(name, other.name)
        && salary == other.salary
        && Objects.equals(hireDay, other.hireDay);
}
\end{lstlisting}

\subsection*{Ereditarietà e \texttt{equals()}}
Quando si estende una classe che ha già un proprio \texttt{equals()}, la sottoclasse deve innanzitutto invocare il metodo della superclasse.

\begin{lstlisting}[language=Java]
public class Manager extends Employee {
    @Override
    public boolean equals(Object otherObject) {
        if (!super.equals(otherObject)) return false;
        Manager other = (Manager) otherObject;
        return bonus == other.bonus;
    }
}
\end{lstlisting}

Si procede in questo modo non solo per una questione di efficienza (richiede meno codice), ma soprattutto perché la superclasse è l'unica che conosce i propri campi (privati), quindi è l'unica che può confrontarli correttamente. La sottoclasse si occupa solo dei campi aggiuntivi di cui è responsabile.  

\subsection*{Uguaglianza ed Ereditarietà: Il Problema della Simmetria}

Il design del metodo \texttt{equals()} deve rispettare i cinque principi stabiliti dalla \textit{Java Language Specification}. La violazione di uno di questi può causare bug imprevedibili quando gli oggetti vengono inseriti in collezioni (come \texttt{HashSet} o \texttt{ArrayList}):
 per ogni riferimento non nullo $x, y, z$:
\begin{itemize}
    \item \textbf{Riflessività}: \texttt{x.equals(x)} deve essere \texttt{true}.
    \item \textbf{Simmetria}: \texttt{x.equals(y)} $\iff$ \texttt{y.equals(x)}.
    \item \textbf{Transitività}: (\texttt{x.equals(y)} $\land$ \texttt{y.equals(z)}) $\implies$ \texttt{x.equals(z)}.
    \item \textbf{Consistenza}: Il risultato non deve cambiare a meno che non cambi lo stato degli oggetti.
    \item \textbf{Test del Null}: \texttt{x.equals(null)} deve sempre essere \texttt{false}.
\end{itemize}

Vale la pena sapere che esistono due scuole di pensiero riguardo l'implementazione del metodo \texttt{equals()} quando si deve gestire il confronto tra una superclasse (\texttt{Employee}) e una sottoclasse (\texttt{Manager}).

\subsubsection*{L'approccio \texttt{instanceof}}
Molti programmatori usano \texttt{if (!(otherObject instanceof Employee))}. 
\begin{itemize}
    \item \textbf{Pro}: Rispetta il \textit{Principio di Sostituzione di Liskov} (in pratica: posso usare un Manager ovunque sia richiesto un Employee).
    \item \textbf{Contro}: \textbf{viola la simmetria}. 
    Se \texttt{Employee.equals} usa \texttt{instanceof}, allora \texttt{e.equals(m)} sarà \texttt{true}. Per la regola di simmetria, anche \texttt{m.equals(e)} deve essere \texttt{true}. Tuttavia, un \texttt{Manager} ha campi extra (il bonus): se per essere simmetrico deve ignorare il bonus quando parla con un \texttt{Employee}, l'uguaglianza diventa logicamente inconsistente per la sottoclasse.
\end{itemize}

\subsubsection*{L'approccio \texttt{getClass()}}
È quello usato nell'esempio di \texttt{Employee}.
\begin{itemize}
    \item \textbf{Pro}: Garantisce la simmetria assoluta. Se le classi non sono identiche, il test fallisce subito.
    \item \textbf{Contro}: Impedisce l'uguaglianza tra tipi diversi anche se condividono la stessa logica di identità.
\end{itemize}



\subsubsection*{La Soluzione di Horstmann: Due Scenari}
Non esiste una regola unica, ma due scenari di design:

\begin{enumerate}
    \item \textbf{Scenario 1: Le sottoclassi aggiungono stato} \\
    Se un \texttt{Manager} ha campi che lo rendono "diverso" da un \texttt{Employee} (come il \texttt{bonus}), l'uguaglianza deve essere limitata alla classe esatta. \\
    $\rightarrow$ \textbf{Usa \texttt{getClass()}}.
    
    \item \textbf{Scenario 2: L'uguaglianza è fissata dalla superclasse} \\
    Se l'uguaglianza è determinata solo da un ID univoco definito nella superclasse e vale per tutti i figli. \\
    $\rightarrow$ \textbf{Usa \texttt{instanceof}} e dichiara il metodo \texttt{equals} come \textbf{\texttt{final}} nella superclasse (per impedire ai figli di romperne la simmetria).
\end{enumerate}

\begin{observation}[Pasticci nella libreria standard]
    La classe \texttt{java.sql.Timestamp} estende \texttt{java.util.Date}. Poiché \texttt{Date} usa \texttt{instanceof}, gli sviluppatori Java hanno ammesso che è impossibile per \texttt{Timestamp} essere simmetrico e accurato allo stesso tempo. È un classico esempio di cattivo design dell'ereditarietà.
\end{observation}

\begin{observation}
    Nel mondo professionale (specialmente con framework come Hibernate/Spring Data), l'uguaglianza basata sull'ID è lo standard quasi assoluto perché rispecchia la struttura dei database.
\end{observation}

\subsection*{ Riepilogo: La Ricetta Definitiva per un metodo \texttt{equals()} perfetto}

Possiamo infine sintetizzare la creazione di un metodo \texttt{equals()} robusto in cinque passaggi fondamentali.

\begin{enumerate}
    \item \textbf{Assegnazione dei nomi}: Chiamare il parametro esplicito \texttt{otherObject}. In seguito, verrà convertito in una variabile locale chiamata \texttt{other}.
    
    \item \textbf{Test di Identità}:
    \begin{lstlisting}[language=Java]
if (this == otherObject) return true;
    \end{lstlisting}

    \item \textbf{Test del \texttt{null}}:
    \begin{lstlisting}[language=Java]
if (otherObject == null) return false;
    \end{lstlisting}
    Questo controllo è obbligatorio per contratto.

    \item \textbf{Il bivio del Test dei Tipi}:
    \begin{itemize}
        \item Se la semantica dell'uguaglianza \textbf{può cambiare} nelle sottoclassi:
        \begin{lstlisting}[language=Java]
if (getClass() != otherObject.getClass()) return false;
ClassName other = (ClassName) otherObject;
        \end{lstlisting}
        \item Se la semantica è \textbf{fissata nella superclasse} (uguaglianza per ID):
        \begin{lstlisting}[language=Java]
if (!(otherObject instanceof ClassName other)) return false;
        \end{lstlisting}
        \textit{Nota: ricordando quanto detto nel paragrafo \ref{instanceof}, la variabile \texttt{other} viene creata e castata automaticamente.}
    \end{itemize}

    \item \textbf{Confronto dei campi}:
    \begin{itemize}
        \item Usa \texttt{==} per i tipi primitivi.
        \item Usa \texttt{Objects.equals(a, b)} per i campi oggetto \\ (per evitare \texttt{NullPointerException}).
    \end{itemize}
    \begin{lstlisting}[language=Java]
return field1 == other.field1 
    && Objects.equals(field2, other.field2) 
    && ...;
    \end{lstlisting}
\end{enumerate}

\begin{important}[Sottoclassi]
    Se stai ridefinendo \texttt{equals()} in una sottoclasse, la prima operazione dopo i controlli di base deve essere l'invocazione di del metodo della superclasse \textbf{\texttt{super.equals(otherObject)}}. Va anche ricordato che il parametro del metodo \texttt{equals()} è sempre di tipo \texttt{Object} (ma ci si protegge da questo tipo di errore usando l'annotazione \texttt{@Override}).
\end{important}

\subsection{Il Metodo \texttt{hashCode()}}

Come approfondito nella sezione teorica in appendice (cfr. \ref{app:hashing}), l'hashing è un processo fondamentale per la sintesi dei dati. In Java, il metodo \texttt{hashCode()} della classe \texttt{Object} restituisce un valore intero (\texttt{int}) che funge da "impronta digitale" dell'oggetto.

L'intero restituito da \texttt{hashCode()} è calcolato in modo da minimizzare la probabilità che due oggetti diversi abbiano lo stesso valore. Mentre l'implementazione predefinita di \texttt{Object} deriva questo numero dall'indirizzo di memoria dell'istanza (identità), nelle classi che rappresentano dati esso viene sovrascritto per riflettere lo stato dell'oggetto.


In Java, l'utilizzo principale di questo metodo avviene all'interno delle \textbf{Collezioni} (come \texttt{HashSet} e \texttt{HashMap}), dove l'hash viene usato come indice per distribuire gli oggetti in diversi "compartimenti" (\textit{buckets}), rendendo le operazioni di ricerca quasi istantanee.Riprenderemo questo concetto nel capitolo sulle collections, ma è importante capire fin da ora che un'implementazione errata di \texttt{hashCode()} può portare a prestazioni disastrose e a comportamenti imprevedibili quando gli oggetti vengono inseriti in queste strutture dati.

Esiste una regola fondamentale che ogni sviluppatore Java deve rispettare: \textbf{se due oggetti sono uguali secondo il metodo \texttt{equals()}, devono necessariamente avere lo stesso \texttt{hashCode()}.}

\begin{important}
    Se si sovrascrive \texttt{equals()}, si deve sovrascrivere anche \texttt{hashCode()}. Se non lo si fa, gli oggetti della classe non funzioneranno correttamente nelle strutture dati basate su hash, poiché oggetti logicamente identici verrebbero cercati in posizioni diverse della memoria.
\end{important}

Dalla versione 7 di Java, il modo più semplice e sicuro per implementare questo metodo è l'utilizzo della funzione statica \texttt{Objects.hash()}. Questa funzione accetta un numero variabile di argomenti e gestisce automaticamente i valori \texttt{null}.

\begin{lstlisting}[language=Java, caption={Esempio di implementazione coordinata in Employee}]
public class Employee {
    private String name;
    private double salary;
    private LocalDate hireDay;

    @Override
    public boolean equals(Object otherObject) {
        // ... (implementazione vista in precedenza) ...
    }

    @Override
    public int hashCode() {
        // Genera l'hash combinando i campi significativi
        return Objects.hash(name, salary, hireDay);
    }
}
\end{lstlisting}

\begin{important}
    Se la classe contiene dei campi che sono \textbf{array}, \texttt{Objects.hash()} non ne verificherà il contenuto (userebbe l'hash dell'indirizzo di memoria dell'array). In questo caso specifico, è necessario utilizzare il metodo \texttt{Arrays.hashCode()} per calcolare un hash basato sui singoli elementi dell'array. Per lo stesso motivo, usare \texttt{Arrays.deepHashCode} per array multidimensionali.
\end{important}

\subsubsection*{Approfondimento: Calcolo manuale e Performance}

Sebbene \texttt{Objects.hash()} sia la scelta standard, in scenari di \textbf{alta performance} o in sistemi legacy, si preferisce il calcolo manuale. Il metodo \texttt{Objects.hash}, infatti, essendo un metodo \textit{varargs}, crea un array temporaneo a ogni invocazione, introducendo un overhead che può diventare significativo in cicli critici.

\begin{lstlisting}[language=Java, caption={Calcolo manuale ottimizzato}]
@Override
public int hashCode() {
    // Si usano numeri primi per distribuire meglio i valori
    return 31 * Objects.hashCode(name) 
         + 11 * Double.hashCode(salary) 
         + 13 * Objects.hashCode(hireDay);
}
\end{lstlisting}

Se il range dei valori dei campi è limitato e noto, è possibile progettare una funzione di hash \textbf{perfetta} (priva di collisioni). 
Ad esempio, per una data composta da giorno, mese e anno, una formula come:
\begin{center}
    \texttt{hash = 31 * 12 * anno + 31 * mese + giorno}
\end{center}
garantisce che ogni data possibile produca un numero univoco (entro un range ragionevole di anni), poiché i "pesi" assegnati ai campi impediscono sovrapposizioni.

\begin{caution}
    Nel calcolo manuale, per i tipi primitivi (\texttt{double}, \texttt{long}), è fondamentale usare i metodi statici delle classi wrapper (es. \texttt{Double.hashCode(salary)}) invece di castare a \texttt{int}, per non perdere precisione e mantenere una buona distribuzione.
\end{caution}

\subsection{Il Metodo \texttt{toString()}}

Il metodo \texttt{toString()} restituisce una stringa che rappresenta il valore dell'oggetto. La sua implementazione predefinita nella classe \texttt{Object} fornisce solo il nome della classe seguito dall'hash code (es. \texttt{Employee@15db9742}), un'informazione raramente utile.

Sovrascrivere questo metodo è considerata una \textit{best practice} fondamentale per facilitare il debugging e il logging, poiché permette di ispezionare lo stato dell'oggetto in modo immediato.

\subsubsection*{Convenzione di Formattazione}
Horstmann raccomanda un formato standard per la stringa restituita:
\begin{center}
    \texttt{NomeClasse[campo1=valore1, campo2=valore2, ...]}
\end{center}

\begin{lstlisting}[language=Java, caption={Implementazione di toString in Employee e Manager}]
// In Employee
public String toString() {
    return getClass().getName() + "[name=" + name + ",salary=" + salary + ",hireDay=" + hireDay + "]";
}

// In Manager (Sottoclasse)
public String toString() {
    return super.toString() + "[bonus=" + bonus + "]";
}
\end{lstlisting}



\subsubsection*{Invocazione Implicita}
Un aspetto potente di \texttt{toString()} è che viene invocato \textbf{automaticamente} dal compilatore in due casi comuni:
\begin{enumerate}
    \item Quando un oggetto viene concatenato a una stringa: \texttt{"Dati: " + myObject}.
    \item Quando un oggetto viene passato a \texttt{System.out.println(myObject)}.
\end{enumerate}

\begin{observation}[Array e toString]
    Gli array ereditano il \texttt{toString} standard di \texttt{Object}, rendendo la loro stampa diretta illeggibile. Per ottenere il contenuto di un array, è necessario utilizzare i metodi statici:
    \begin{itemize}
        \item \texttt{Arrays.toString(a)}: per array monodimensionali.
        \item \texttt{Arrays.deepToString(a)}: per array multidimensionali.
    \end{itemize}
\end{observation}

\subsection{Il caso particolare dei Record}

Nonostante la loro sintassi semplificata, i Record rimangono pienamente inseriti nella gerarchia degli oggetti Java.

\begin{itemize}
    \item \textbf{Ereditarietà da \texttt{Object}}: Ogni record è, a tutti gli effetti, un \texttt{Object}. Puo essere assegnato a una variabile di tipo \texttt{Object} e ne possiede tutti i metodi fondamentali.
    \item \textbf{La classe \texttt{java.lang.Record}}: Ogni record estende implicitamente la classe astratta \texttt{java.lang.Record}. Questa classe funge da "ponte" tra il tipo specifico e la superclasse cosmica \texttt{Object}.
\end{itemize} \vspace{\baselineskip}

Proprio perché estendono implicitamente \texttt{java.lang.Record}:
\begin{enumerate}
    \item Un record \textbf{non puo estendere} altre classi.
    \item Un record \textbf{è sempre \texttt{final}}, il che significa che nessuna classe può estendere un record.
\end{enumerate} \vspace{\baselineskip}

Essendo progettati per essere "vettori di dati immutabili", per i Records Java rimuove tutto il \textit{boilerplate code} relativo ai metodi della classe \texttt{Object}: per ogni Record, il compilatore genera automaticamente le seguenti implementazioni:
    \begin{itemize}
        \item \textbf{\texttt{equals()}}: Confronta due record componente per componente (state-based equality). Due record sono uguali se hanno lo stesso tipo e i valori di tutti i loro campi coincidono.
        \item \textbf{\texttt{hashCode()}}: Generato in modo coerente con \texttt{equals()}.
        \item \textbf{\texttt{toString()}}: Restituisce una stringa leggibile che elenca tutti i componenti e i loro valori (es. \texttt{Point[x=10.0, y=20.0]}).
    \end{itemize}

\begin{observation}
    Sebbene sia possibile effettuare l'override di \texttt{equals()} anche in un record, ciò ne violerebbe quasi sempre la semantica. I record dovrebbero rappresentare \textbf{solo} i dati che contengono; pertanto, l'uguaglianza basata sullo stato generata automaticamente è quasi sempre quella corretta.
\end{observation}
