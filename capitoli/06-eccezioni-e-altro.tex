\chapter{Eccezioni, Asserzioni e Logging}

\section{Gestione delle Eccezioni}

Il passaggio dalla programmazione puramente didattica a quella professionale coincide con la consapevolezza che il codice deve operare in un ambiente ostile (dati errati, bug, risorse esterne indisponibili).

In presenza di un'anomalia, il software deve garantire tre requisiti minimi:
\begin{enumerate}
    \item \textbf{Notifica}: Informare l'utilizzatore del problema occorso.
    \item \textbf{Persistenza}: Salvare lo stato corrente per evitare la perdita di dati.
    \item \textbf{Terminazione Controllata}: Consentire una chiusura del programma che rilasci correttamente le risorse impegnate.
\end{enumerate}
\vspace{\baselineskip}

Storicamente, i metodi segnalavano il fallimento tramite valori sentinella (\texttt{null}, \texttt{-1}, \texttt{false}). Questo approccio è oggi considerato \textbf{pericoloso} e \textbf{poco informativo}.

\vspace{\baselineskip}

In Java, un'anomalia durante l'esecuzione (runtime) non provoca necessariamente il crash immediato del sistema operativo, ma attiva un meccanismo strutturato chiamato \textbf{Gestione delle Eccezioni} che si basa sull'uso di classi preposte. 

Illustriamo questo meccanismo in un caso specifico. Supponiamo che il processore incontri l'istruzione \texttt{int x = 10 / 0}; la Java Virtual Machine interrompe il flusso lineare e segue i seguenti passaggi:

\begin{enumerate}
    \item \textbf{Creazione dell'Oggetto Eccezione}: Non appena si verifica l'errore, la JVM crea istantaneamente un oggetto di una classe specifica. Nel caso della divisione per zero, viene istanziata la classe:
    \begin{center}
        \texttt{java.lang.ArithmeticException}
    \end{center}
    Questo oggetto contiene:
    \begin{itemize}
        \item Il \textbf{tipo} di eccezione.
        \item Il \textbf{messaggio di errore} (es. "/ by zero").
        \item Lo \textbf{Stack Trace}: una fotografia dei metodi che erano in esecuzione al momento del guasto.
    \end{itemize}
    \item \textbf{Lancio (Throwing)}: Il metodo in cui si è verificato l'errore "lancia" (\textit{throws}) l'oggetto eccezione al sistema di runtime. L'esecuzione del codice normale nel metodo corrente viene immediatamente sospesa.
    \item \textbf{Ricerca del Gestore (Catching)}: Il sistema di runtime cerca nel \textbf{Call Stack} (lo stack delle chiamate ai metodi) un blocco di codice in grado di gestire l'eccezione.
    \begin{enumerate}
        \item Il sistema controlla se il punto dell'errore è gestito da un apposito blocco di codice (\texttt{try}) .
        \item In caso positivo, verifica se esiste un blocco \texttt{catch} associato che sia compatibile con il tipo di eccezione lanciata; se lo trova, l'esecuzione continua: viene eseguito il blocco \texttt{catch}.
        \item Se non viene trovato nel metodo corrente, il sistema passa al metodo "chiamante" e così via, risalendo la gerarchia.
    \end{enumerate}
    \item \textbf{Il Default Exception Handler}: Se la ricerca arriva al metodo \texttt{main} senza aver trovato alcun gestore, l'eccezione viene catturata dal \textbf{Default Exception Handler}. Questo esegue due azioni finali:
    \begin{itemize}
        \item Stampa a video lo Stack Trace (il testo rosso in console).
        \item Termina forzatamente il thread in cui si è verificata l'eccezione.
    \end{itemize}
\end{enumerate}
\vspace{\baselineskip}


Ecco un confronto tra un codice non gestito (che causa il crash) e uno gestito correttamente. Nella seconda parte è anche illustrata la sintassi del \texttt{try-catch} in una forma elementare. 

\begin{lstlisting}[caption=Esempio di crash]
public class Test {
    public static void main(String[] args) {
        int a = 10;
        int b = 0;
        int result = a / b; // Il programma termina qui
        System.out.println("Risultato: " + result); 
    }
}
\end{lstlisting}

\begin{lstlisting}[caption=Gestione dell'anomalia e sintassi base del \texttt{try-catch}]
public class Test {
    public static void main(String[] args) {
        try {
            int result = 10 / 0;
        } catch (ArithmeticException e) {
            System.err.println("Errore: Impossibile dividere per zero!");
        }
        System.out.println("Il programma continua l'esecuzione...");
    }
}
\end{lstlisting}

La gestione delle eccezioni trasforma dunque un errore fatale in un evento che può essere previsto e mitigato, permettendo al software di rimanere robusto e di fornire feedback utili all'utente o allo sviluppatore tramite lo Stack Trace.
\vspace{\baselineskip}

Confrontiamo l'evoluzione delle API Java per capire il vantaggio delle eccezioni:

\begin{itemize}
    \item \textbf{Vecchio Stile (Legacy)}: \texttt{file.delete()} restituisce \texttt{false} se fallisce. 
    \begin{itemize}
        \item \textit{Problema}: Perché e' fallito? Permessi negati? File inesistente? Disco protetto? Non lo sapremo mai.
    \end{itemize}
    \item \textbf{Nuovo Stile (NIO)}: \texttt{Files.delete(path)} lancia un'eccezione, ad esempio: 
    \begin{center}
        \texttt{NoSuchFileException}, \texttt{AccessDeniedException}, ecc...
    \end{center}
    
    \begin{itemize}
        \item \textit{Vantaggio}: L'errore è auto-esplicativo e costringe il programmatore a decidere come gestirlo.
    \end{itemize}
\end{itemize}
\vspace{\baselineskip}

È fondamentale capire che in Java la differenza tra le eccezioni non è solo gerarchica, ma riguarda soprattutto il modo in cui il compilatore impone di gestirle. La distinzione principale è tra \textit{checked} e \textit{unchecked}:

\begin{itemize}
    \item \textbf{Checked}: il compilatore obbliga il programmatore a gestirle (vedremo a breve in dettaglio cosa questo voglia dire). In genere rappresentano problemi esterni o situazioni recuperabili, come errori di I/O, rete o database.
    
    \item \textbf{Unchecked}: il compilatore non obbliga a gestirle né a dichiararle. Sono spesso dovute a errori logici del programmatore (ad esempio \texttt{NullPointerException}, indici fuori range, argomenti non validi), oppure guasti gravi della JVM o del sistema.
\end{itemize}

\vspace{\baselineskip}

Tutti i \texttt{Throwable} che derivano da \texttt{Error} o da \texttt{RuntimeException} sono unchecked, tutti gli altri sono \textit{checked exceptions}.

\vspace{\baselineskip}

Possiamo liquidare rapidamente il ramo della gerarchia relativa agli \texttt{Error}: si tratta di errori talmente critici (come errori interni e situazioni di esaurimento delle risorse all'interno del sistema di runtime Java) che non si dovrebbe mai tentare di gestire. In questi casi, il programmatore ha poche possibilità di intervento: l'unica azione sensata è solitamente informare l'utente e terminare il programma in modo consono. Fortunatamente, queste situazioni sono abbastanza rare nella pratica.

Da questo momento in poi ci occuperemo quindi del ramo \texttt{Exception} della gerarchia, ovvero della classe dei problemi gestibili.

\subsection{Dichiarare e Lanciare Eccezioni}

Un metodo Java può terminare in due modi: restituendo un valore o "esplodendo" con un'eccezione. Se un metodo non è in grado di gestire una situazione anomala, deve dichiarare questa possibilità al mondo esterno. La sintassi per dichiarare e \textit{lanciare} un'eccezione è illustrata nel seguente esempio elementare: 

\begin{lstlisting}[caption={Esempio di metodo che lancia un'eccezione}]
public void setAge(int age) throws IllegalArgumentException {
    if (age < 0) {
        // Creiamo l'oggetto eccezione e lo lanciamo
        throw new IllegalArgumentException("age cannot be negative: " + age);
    }
    this.age = age;
}
\end{lstlisting}

È fondamentale non confondere queste due parole chiave, simili ma con scopi diversi:
\begin{itemize}
    \item \textbf{\texttt{throws}}: Si usa nella \textbf{firma del metodo} per avvertire il compilatore e il chiamante che quel metodo può lanciare determinate eccezioni.
    \item \textbf{\texttt{throw}}: Si usa nel \textbf{corpo del metodo} per lanciare effettivamente l'oggetto eccezione.
\end{itemize}
\vspace{\baselineskip}
Quando viene eseguita l'istruzione \texttt{throw}:
\begin{enumerate}
    \item L'esecuzione del metodo si interrompe \textbf{immediatamente}.
    \item Le istruzioni successive (come l'assegnamento \texttt{this.age = age}) non vengono eseguite.
    \item Il controllo passa al chiamante, cercando il primo blocco \texttt{try-catch} disponibile che sappia gestire \texttt{IllegalArgumentException}.
\end{enumerate}

\begin{tip}[]
Quando l'eccezione lanciata è \textit{checked}, è \textbf{obbligatorio} dichiararla nella testata del metodo. In questo esempio abbiamo invece usato \texttt{IllegalArgumentException}, che è una \textbf{Unchecked Exception}, quindi la clausola \texttt{throws} nella firma del metodo è in tal caso opzionale.

In generale, intercettare eccezioni \textit{unchecked} (e ancor più gli \texttt{Error}) non dovrebbe essere la strategia ordinaria: spesso tali eccezioni segnalano errori di programmazione o uso scorretto delle API, che andrebbero prevenuti scrivendo codice robusto e validando correttamente gli input. Tuttavia, in alcuni contesti può essere sensato intercettarle per effettuare logging, traduzione dell'errore o gestione centralizzata.

Horstmann riassume efficacemente l'idea con una frase suggestiva: \textit{"se si verifica un'eccezione unchecked, spesso il problema è nel codice, non in un evento imprevedibile"}.
\end{tip}



\begin{observation}
    Ciò che diremo per i metodi varrà anche per i costruttori. Un metodo o un costruttore può lanciare più di un tipo di eccezione; in questo caso basta elencarle dopo lo specificatore \texttt{throws} separate da virgole (se necessario).
\end{observation}

\subsection{Il Blocco \texttt{try-catch}: gestione locale}

Come abbiamo visto, se si verifica un'eccezione che non viene catturata in alcun luogo, il programma termina e stampa un messaggio sulla console (lo \textit{stack trace}). Per evitare questo e permettere al programma di continuare o chiudersi in maniera controllata, usiamo il blocco \texttt{try-catch}.

La struttura base consiste nel racchiudere il codice "pericoloso" in un blocco \texttt{try} e definire la gestione del problema in uno o più blocchi \texttt{catch}.

\begin{lstlisting}[caption={Sintassi base try-catch-finally}]
try {
    // 1. Codice che potrebbe lanciare un'eccezione
    // Se un'eccezione esplode qui, il resto del blocco viene saltato
} catch (TipoEccezione1 e) {
    // 2. Codice eseguito solo se si verifica TipoEccezione1
    // 'e' e' l'oggetto che contiene i dettagli dell'errore
} catch (TipoEccezione2 e) {
    // 3. Si possono avere piu blocchi catch per diversi errori
} finally {
    // 4. (Opzionale) Codice eseguito SEMPRE, 
    // sia che ci sia stata un'eccezione, sia che non ci sia stata.
}
\end{lstlisting}


Il flusso di esecuzione è il seguente:

\begin{enumerate}
    \item \textbf{Se non ci sono errori}: Viene eseguito tutto il blocco \texttt{try}, i blocchi \texttt{catch} vengono saltati, viene eseguito il \texttt{finally} (se presente) e si continua con l'esecuzione del metodo.
    \item \textbf{Se si verifica un errore}: 
    \begin{itemize}
        \item L'esecuzione nel \texttt{try} si ferma alla riga che ha generato l'eccezione.
        \item La JVM cerca un blocco \texttt{catch} compatibile con il tipo di eccezione lanciata.
        \item Se lo trova, esegue il codice nel \texttt{catch}, passa al \texttt{finally} (se presente) e continua con l'esecuzione del metodo. 
        \item Se NON lo trova, viene eseguito l'eventuale \texttt{finally}, il metodo termina e l'eccezione "risale" verso il chiamante.
    \end{itemize}
\end{enumerate}

\begin{warning}
L'ordine dei blocchi \texttt{catch} e' fondamentale. Java esegue il primo \texttt{catch} compatibile che trova. Pertanto, si devono sempre mettere le eccezioni **più specifiche** (es. \texttt{FileNotFoundException}) prima di quelle **più generiche** (es. \texttt{IOException} o \texttt{Exception}). Altrimenti, la più specifica non verrà mai raggiunta.
\end{warning}

A partire da Java 7, se si devono gestire diverse eccezioni nello stesso identico modo, non è più necessario scrivere blocchi \texttt{catch} duplicati: basta usare la sintassi \textit{multi-catch} separando le classi con l'operatore \textit{pipe} (\texttt{|}). Oltre a rendere il codice più leggibile, questa pratica è più efficiente: il compilatore genera un unico blocco di \textit{bytecode} per le eccezioni condivise, riducendo la dimensione del file compilato.
\begin{lstlisting}[caption={Esempio di multi-catch}]
    catch (FileNotFoundException | UnknownHostException e) {
    // Gestione comune per entrambi i problemi
    logger.error("Errore di connessione o file: " + e.getMessage());
}
\end{lstlisting}

\begin{warning}
    Quando si usa il multi-catch con l'operatore | , la variabile dell'eccezione (\texttt{e}) è implicitamente \textbf{final}: non si può riassegnarla all'interno del blocco.

    Un altro dettaglio tecnico importante è che il tipo di \texttt{e} nel multi-catch non è un "tipo unione" (anche quando l'IDE ci dice che è così). Il compilatore assegna a \texttt{e} il tipo della \textbf{superclasse comune più vicina} (\textit{LUD: least upper bound}) tra quelle elencate. In genere questo non è un problema ma, in caso contrario, probabilmente è meglio tornare a usare blocchi \texttt{catch} separati. Il codice risulterà molto più pulito e meno propenso a errori.
\end{warning}

Il blocco \texttt{finally} viene utilizzato quasi esclusivamente per la pulizia delle risorse (chiudere file, connessioni database, socket). La sua caratteristica unica è che viene eseguito \textbf{anche se} nel blocco \texttt{try} o \texttt{catch} era presente un'istruzione \texttt{return}.

L'unica situazione in cui \texttt{finally} potrebbe non essere eseguito è se la JVM si arresta bruscamente (es. \texttt{System.exit(0)}) o se si verifica un \texttt{Error} catastrofico dell'hardware.


\begin{warning}
Sebbene il \texttt{finally} sia fondamentale per la chiusura delle risorse, bisogna stare attenti a un errore comune: inserire istruzioni che alterano il flusso di controllo (come \texttt{return} o \texttt{throw}) all'interno del \texttt{finally}.
Se un blocco \texttt{finally} contiene un \texttt{return}, questo "sovrascriverà" qualsiasi valore di ritorno o eccezione lanciata nei blocchi \texttt{try} o \texttt{catch}. Analogamente per i \texttt{throw}.
\end{warning}

\subsection{La clausola \texttt{throws}: gestione delegata}

Non sempre gestire un'eccezione localmente con un \texttt{try-catch} è la scelta migliore. Spesso, il metodo che rileva l'errore non sa come risolverlo (ad esempio, un metodo di basso livello che legge un file non sa se deve mostrare un messaggio di errore all'utente o riprovare). In questi casi, è preferibile \textbf{propagare} l'eccezione al metodo chiamante.

Per delegare la gestione, si omette il blocco \texttt{try-catch} e si aggiunge la clausola \texttt{throws} nella firma (testata) del metodo. Questo avverte il compilatore che chiunque chiamerà questo metodo dovrà assumersi la responsabilità di gestire l'eventuale "esplosione".

\begin{lstlisting}[caption={Esempio di delega dell'eccezione}]
// Il metodo non gestisce l'errore, ma "avvisa" il chiamante
public void leggiFileConfigurazione(String percorso) throws FileNotFoundException {
    // Se il file non esiste, il costruttore lancia FileNotFoundException
    FileInputStream fis = new FileInputStream(percorso); 
    // ... logica di lettura ...
}
\end{lstlisting}

Questo tipo di delega segue le seguenti regole:
\begin{itemize}
    \item \textbf{Per le Checked Exceptions}: La delega tramite \texttt{throws} è \textbf{obbligatoria} se non si usa un \texttt{try-catch}. Se dimentichi di dichiararla, il codice non compila.
    \item \textbf{Per le Unchecked Exceptions}: La delega è \textbf{facoltativa}.
    \item \textbf{Passaggio di consegne}: Se il metodo A chiama il metodo B (che lancia un'eccezione), il metodo A ha di nuovo le stesse due opzioni: catturarla con un \texttt{try-catch} o dichiararla a sua volta con \texttt{throws} (di nuovo: obbligatorio in caso di eccezione \textit{checked}, opzionale altrimenti), passandola al metodo superiore.
\end{itemize}

\begin{bestpractice}[Best Practice]
Le eccezioni dovrebbero essere intercettate nel livello più basso che sia effettivamente in grado di gestirle in modo corretto; in caso contrario, è preferibile propagarle verso livelli superiori che dispongano del contesto necessario per decidere come reagire.
\end{bestpractice}

Sebbene ormai il meccanismo di gestione locale o delega dovrebbe essere chiaro anche nel caso particolare delle \textit{checked exceptions}, evidenziamo la seguente regola del \textbf{Catch or Declare}:

\begin{tip}[Regola del Catch-or-Declare]
Se un metodo X ne invoca un altro che lancia una checked exception, il codice \textbf{non compila} finché non si soddisfa una di queste due condizioni:
\begin{enumerate}
    \item \textbf{Catch}: Si intercetta l'eccezione con un blocco \texttt{try-catch} per gestirla.
    \item \textbf{Declare}: Si aggiunge la clausola \texttt{throws} alla firma del metodo X per delegare la responsabilità a chiunque lo invochi.
\end{enumerate}
\end{tip}

\subsection{Ereditarietà ed Eccezioni}

Se un metodo della sottoclasse sovrascrive un metodo della superclasse, valgono le seguenti regole:

\begin{itemize}
    \item \textbf{Non può} dichiarare eccezioni checked più generiche di quelle previste nel metodo originale.
    \item \textbf{Non può} aggiungere nuove eccezioni checked incompatibili con la testata del metodo ereditato.
    \item \textbf{Può} dichiarare eccezioni più specifiche (sottoclassi di quelle originali).
    \item \textbf{Può} decidere di non dichiarare alcuna eccezione.
\end{itemize}

In particolare, se il metodo della superclasse non dichiara alcuna eccezione checked, allora il metodo sovrascritto nella sottoclasse \textbf{non può propagarne alcuna che sia checked}. Può comunque usare codice che genera checked exceptions, ma in tal caso deve gestirle localmente mediante \texttt{try-catch}.

Queste restrizioni riguardano solo le \textit{checked exceptions}. Le \textit{unchecked exceptions} possono invece essere lanciate liberamente anche se non vengono lanciate nel metodo della superclasse.

\begin{bestpractice}
Quando progetti una gerarchia di classi, valuta con attenzione quali eccezioni inserire nella clausola \texttt{throws} della superclasse. Una firma troppo restrittiva può rendere più difficile l'implementazione delle sottoclassi che interagiscono con risorse esterne o operazioni fallibili.
\end{bestpractice}

\subsection{Creare Classi di Eccezione Personalizzate}

Sebbene Java offra una vasta gamma di classi di eccezione, potresti imbatterti in problemi che non sono descritti adeguatamente da quelle standard. In questi casi, basta creare un'eccezione personalizzata: è sufficiente estendere la classe \texttt{Exception} (per creare una \textit{checked exception}) o una sua sottoclasse (come \texttt{IOException}). Se invece si desidera una \textit{unchecked exception}, si deve estendere \texttt{RuntimeException}.

Una volta definita, la tua eccezione è pronta per essere lanciata all'interno dei tuoi metodi esattamente come un'eccezione di sistema.

\begin{lstlisting}[caption={Lancio di un'eccezione personalizzata}]
public String leggiDati(Scanner in) throws FileFormatException {
   // ... logica di lettura ...
   while (...) {
      if (ch == -1) { // Fine del file in attesa
         if (n < len)
            throw new FileFormatException("Il file termina inaspettatamente");
      }
      // ...
   }
   return s;
}
\end{lstlisting}

\begin{tip}
Quando estendi \texttt{Exception}, stai creando una \textbf{Checked Exception}: il compilatore obbligherà chiunque usi il tuo metodo a gestirla o a dichiararla. Se vuoi che la tua eccezione sia più "leggera" e non obbligatoria da gestire, estendi \texttt{RuntimeException}.
\end{tip}

\subsection{Exceptions Chaining}

A volte, all'interno di un blocco \texttt{catch}, potresti voler lanciare una nuova eccezione. Questo si fa solitamente per "astrarre" il problema: chi usa il tuo codice non deve conoscere i dettagli tecnici (come un errore del database), ma deve sapere che il tuo sottosistema ha avuto un problema.

Invece di lanciare una nuova eccezione perdendo traccia di quella originale, la pratica migliore è \textbf{incatenarle (chaining)}. In questo modo, l'eccezione originale viene salvata come "causa" della nuova.

\begin{lstlisting}[caption={Esempio di Exception Chaining}]
try {
    // Tentativo di prelievo dal database
    eseguiQueryPrelievo(); 
} catch (SQLException originale) {
    // Creiamo una nuova eccezione di alto livello
    // Passiamo l'eccezione originale come "causa" (secondo parametro)
    var e = new BankServiceException("Errore durante il prelievo", originale);
    throw e;
}
\end{lstlisting}

\begin{tip}
Quasi tutte le classi di eccezione standard hanno un costruttore che accetta la "causa". Se una classe non lo prevede, puoi usare il metodo \texttt{e.initCause(originale)} prima di lanciarla. Chi riceve l'eccezione potrà poi risalire all'errore iniziale con \texttt{e.getCause()}.
\end{tip}

Un altro uso comune di questa tecnica è quando ti trovi in un metodo che \textbf{non può} lanciare checked exceptions (ad esempio a causa di un override), ma il codice al suo interno ne genera una.

\begin{lstlisting}[caption={Da Checked a Unchecked}]
public void stampaDocumento() { // Non dichiara throws
    try {
        generaPDF(); // Questo metodo lancia IOException (checked)
    } catch (IOException e) {
        // La impacchettiamo in una RuntimeException (unchecked)
        // Cosi il compilatore e' soddisfatto e non perdiamo l'errore
        throw new RuntimeException("Fallimento critico durante la stampa", e);
    }
}
\end{lstlisting}

\begin{bestpractice}
L'incatenamento è caldamente raccomandato. Permette di mantenere un'interfaccia pulita verso l'alto (eccezioni di alto livello) senza sacrificare i dettagli tecnici necessari per il debugging (lo \textit{stack trace} mostrerà tutta la catena delle cause).
\end{bestpractice}

\subsection{\texttt{try-with-resources}}

La chiusura delle risorse (file, connessioni, socket) è una delle operazioni più critiche. Se una risorsa non viene chiusa, si rischia un \textit{resource leak} che può bloccare il sistema.

È possibile usare il blocco \texttt{finally} anche senza il \texttt{catch}. Questo serve a garantire che una risorsa venga chiusa indipendentemente dal fatto che si verifichi un'eccezione o meno.

\begin{lstlisting}[caption={Utilizzo di try-finally per la chiusura risorse}]
InputStream in = ...;
try {
    // codice che potrebbe lanciare eccezioni
} finally {
    in.close(); // Eseguito sempre
}
\end{lstlisting}

Per una gestione perfetta, è preferibile separare le responsabilità usando blocchi \texttt{try} annidati: uno interno per garantire la chiusura e uno esterno per gestire/segnalare gli errori.
\begin{lstlisting}[caption={un tipico pattern di \texttt{try} annidati}]
InputStream in = . . .;
try
{
   try
   {
      code that might throw exceptions
   }
   finally
   {
      in.close();
   }
}
catch (IOException e)
{
   show error message
}
\end{lstlisting}

Per evitare la complessità dei blocchi annidati, esiste il \texttt{try-with-resources}. Questa sintassi può essere usata con qualsiasi classe che implementi l'interfaccia \textbf{\texttt{AutoCloseable}} (o la sua sottointerfaccia \texttt{Closeable}); essa ha un unico metodo \texttt{void close()} che può lanciare una \texttt{IOException}.

\begin{lstlisting}[caption={Sintassi del try-with-resources}]
// La risorsa viene dichiarata tra le parentesi del try
try (var in = new Scanner(Path.of("input.txt"))) {
    while (in.hasNext())
        System.out.println(in.next());
} // in.close() viene chiamato AUTOMATICAMENTE qui
\end{lstlisting}

\begin{tip}
Puoi gestire più risorse contemporaneamente separandole con un punto e virgola:
\texttt{try (var in = ...; var out = ...) \{ ... \}}. Verranno chiuse tutte in ordine inverso rispetto alla dichiarazione.
\end{tip}

\begin{tip}
Usa sempre il \texttt{try-with-resources} quando lavori con oggetti che richiedono una chiusura. Scrivere manualmente la logica di chiusura con i \texttt{finally} annidati è complicato e porta spesso a bug silenziosi.
\end{tip}

Un blocco \texttt{try-with-resources} può avere i propri blocchi \texttt{catch} e \texttt{finally}. L'ordine di esecuzione è:
\begin{enumerate}
    \item Viene eseguito il corpo del \texttt{try}.
    \item Vengono chiuse le risorse dichiarate nell'intestazione.
    \item Vengono eseguiti eventuali blocchi \texttt{catch} e il blocco \texttt{finally}.
\end{enumerate}
Questo significa che i \texttt{catch} possono catturare anche le eccezioni avvenute durante la chiusura automatica delle risorse.

\subsubsection*{Eccezioni Soppresse}
Uno dei vantaggi principali del \texttt{try-with-resources} è come gestisce il conflitto tra eccezioni. Come abbiamo visto, nel \texttt{finally} classico un'eccezione durante la chiusura "nascondeva" quella principale.

Anche quando usiamo il \texttt{try-with-resources}, può capitare un "conflitto di eccezioni": un errore durante l'esecuzione del codice (\textit{primary exception}) e uno o più errori durante la chiusura automatica delle risorse (\textit{suppressed exceptions}). Il costrutto però gestisce in modo automatico il conflitto. 

Un'eccezione si dice \textbf{soppressa} quando viene catturata silenziosamente dalla JVM e allegata all'eccezione principale invece di essere lanciata autonomamente. Questo evita che l'errore di chiusura "mascheri" l'errore originale che ha causato il fallimento del programma.

Con il \texttt{try-with-resources}, se sia il blocco \texttt{try} che il metodo \texttt{close()} lanciano un'eccezione:
\begin{enumerate}
    \item L'eccezione originale del \texttt{try} viene lanciata regolarmente.
    \item Le eccezioni lanciate dal metodo \texttt{close()} vengono "soppresse" e aggiunte a quella principale.
    \item È possibile recuperarle in un secondo momento usando il metodo \texttt{e.getSuppressed()}.
\end{enumerate}

\begin{lstlisting}[caption={Recupero delle eccezioni soppresse}]
try (var out = new PrintWriter("file.txt")) {
    throw new IllegalStateException("Errore nel corpo del try");
} catch (IllegalStateException e) {
    System.err.println("Errore primario: " + e.getMessage());
    
    // Recuperiamo l'array delle eccezioni "nascoste"
    Throwable[] suppressed = e.getSuppressed();
    for (Throwable t : suppressed) {
        System.err.println("Eccezione soppressa: " + t.getMessage());
    }
}
\end{lstlisting}

\begin{tip}
Non devi preoccuparti di gestire manualmente le eccezioni soppresse nel 99\% dei casi. La cosa fondamentale è che, guardando lo \textit{stack trace} nella console, Java te le mostrerà automaticamente sotto la dicitura \texttt{Suppressed: ...}, permettendoti di vedere l'intera catena del disastro.
\end{tip}

\subsection{Costruttori e Metodi di Eccezioni}

Ogni volta che scrivi \texttt{catch (Exception e)}, la variabile \texttt{e} è un riferimento a un oggetto. Poiché le eccezioni ereditano da \texttt{java.lang.Throwable}, ereditano un set di metodi preziosi per la diagnostica.

\begin{itemize}
    \item \textbf{\texttt{String getMessage()}}: Restituisce la stringa descrittiva passata al costruttore. È il "cosa è successo" in formato leggibile.
    \item \textbf{\texttt{Throwable getCause()}}: Restituisce l'eccezione interna che ha scatenato quella attuale (fondamentale per l'\textit{incatenamento} visto nei paragrafi precedenti).
    \item \textbf{\texttt{StackTraceElement[] getStackTrace()}}: Permette di accedere programmaticamente alla "pila" delle chiamate. Ogni elemento contiene il nome del file, la classe, il metodo e la riga di codice.
    \item \textbf{\texttt{void printStackTrace()}}: Stampa l'intera traccia dell'errore sullo standard error.
    \item \textbf{\texttt{void addSuppressed(Throwable t)}}: Aggiunge un'eccezione alla lista delle soppresse (usato automaticamente dal try-with-resources).
\end{itemize}
\vspace{\baselineskip}

Per quanto riguarda i costruttori, creando una tua eccezione, dovresti quasi sempre implementare questi quattro per garantire massima flessibilità:

\begin{lstlisting}[caption={I 4 costruttori "canonici" di un'eccezione}]
public class MiaException extends Exception {
    // 1. Vuoto: per errori generici senza dettagli
    public MiaException() { super(); }

    // 2. Con messaggio: per spiegare cosa e' andato storto
    public MiaException(String message) { super(message); }

    // 3. Con causa: per "impacchettare" un altro errore
    public MiaException(Throwable cause) { super(cause); }

    // 4. Con messaggio e causa:
    public MiaException(String message, Throwable cause) {
        super(message, cause);
    }
}
\end{lstlisting}

\begin{tip}
Evita di usare \texttt{e.printStackTrace()} nel codice di produzione backend. Questo metodo scrive direttamente sulla console del server, che spesso viene ignorata o sovrascritta. Usa sempre un logger (\texttt{logger.log(Level.SEVERE, "msg", e)}) che invece salva lo stack trace in modo ordinato su file.
\end{tip}

\section{Le Asserzioni (Assertions)}

Le asserzioni servono a verificare presupposti che il programmatore ritiene debbano essere sempre veri (\textit{invarianti}) in un determinato punto del codice. 

Java mette a disposizione la parola chiave \texttt{assert} con due forme:

\begin{enumerate}
    \item \texttt{assert condizione;}
    \item \texttt{assert condizione : espressione;}
\end{enumerate}

Se la \texttt{condizione} è \texttt{false}, viene lanciato un \texttt{AssertionError}. Nella seconda forma, l'\texttt{espressione} viene convertita in stringa e usata come messaggio di errore.

\begin{lstlisting}[caption={Esempio di utilizzo delle asserzioni}]
double y = Math.sqrt(x);
// Sono certo che x non sia negativo per logica del programma
assert x >= 0 : "x era negativo: " + x;
\end{lstlisting}

\begin{tip}[]
L'unico scopo dell'espressione dopo i due punti è fornire dettagli per il debugging. Non cercare di usarla per "recuperare" l'errore: le asserzioni servono a far fallire il programma subito se qualcosa di "impossibile" accade. È un'applicazione del principio di \textit{Fail Fast}: è molto meglio che un programma vada in crash subito durante i test, piuttosto che continui a girare con dati corrotti, rischiando di salvare informazioni sbagliate nel database o generare bug "silenziosi" difficilissimi da scovare mesi dopo.
\end{tip}

Perché usare \texttt{assert} invece di \texttt{if-throw}? A differenza delle normali istruzioni di controllo, le asserzioni hanno un vantaggio prestazionale unico:

\begin{itemize}
    \item \textbf{In Sviluppo}: Vengono abilitate per trovare bug logici.
    \item \textbf{In Produzione}: Vengono disabilitate di default. Il bytecode relativo alle asserzioni non viene nemmeno eseguito, garantendo che il programma sia il più veloce possibile.
\end{itemize}
\medskip

\subsection{Attivare/Disattivare Asserzioni}
Le asserzioni sono dunque disabilitate di deafult; per abilitarle, bisogna inserire i parametri \texttt{-enableassertions} (oppure la forma abbreviata \texttt{-ea}) :

\begin{lstlisting}[language=bash, caption={}]
java -enableassertions MyApp
\end{lstlisting}

Java permette di filtrare l'attivazione delle asserzioni per singola classe o per pacchetto:

\begin{itemize}
    \item \textbf{Singola Classe:} \texttt{-ea:NomeClasse} attiva le asserzioni solo per quella classe.
    \item \textbf{Pacchetto e Sottopacchetti:} \texttt{-ea:com.azienda.libreria...} attiva le asserzioni per tutte le classi del pacchetto specificato e di tutti i suoi sottopacchetti.
    \item \textbf{Unnamed Package:} L'opzione \texttt{-ea:...} attiva le asserzioni per tutte le classi che non appartengono a un pacchetto dichiarato (il cosiddetto \textit{unnamed package}).
\end{itemize}
\medskip
Allo stesso modo, è possibile disabilitare le asserzioni in punti specifici del codice utilizzando il flag \texttt{-disableassertions} (o \texttt{-da}). Questo è particolarmente utile quando si vuole monitorare tutto il progetto tranne una classe o un modulo specifico che sappiamo essere già stabile o che genera troppi errori di asserzione irrilevanti.

\begin{lstlisting}[language=bash, caption={Esempio di abilitazione specifica}]
# 1. Abilitare le asserzioni per una SINGOLA CLASSE
java -ea:com.azienda.progetto.ValidatoreDati MyApp

# 2. Abilitare le asserzioni per un SINGOLO PACKAGE (e i suoi sottopacchetti)
# Nota: i tre punti (...) sono fondamentali
java -ea:com.azienda.progetto.servizi... MyApp

# 3. Abilitare in un PACKAGE, ma disabilitarle in una CLASSE specifica
# Utile se quella classe e' vecchia (legacy) e lancia errori che vuoi ignorare
java -ea:com.azienda.progetto.logica... -da:com.azienda.progetto.logica.VecchioAlgoritmo MyApp

# 4. Abilitare in un PACKAGE, ma disabilitarle in un suo SOTTOPACKAGE
# Utile per testare la parte stabile e ignorare quella sperimentale/instabile
java -ea:com.azienda.core... -da:com.azienda.core.sperimentale... MyApp

# Attiva le asserzioni nell'unnamed package, ma disabilitale in MyClass
java -ea:... -da:MyClass MyApp
\end{lstlisting}

È possibile combinare più flag. In caso di conflitto, la regola è: \textbf{il filtro più specifico vince}.

\begin{lstlisting}[caption={Esempio di configurazione mista}]
# Abilita tutto il progetto, ma tieni spente le asserzioni nel pacchetto "legacy"
java -ea -da:com.vecchio.codice... MyApp

# Abilita solo una classe specifica mentre tutto il resto e' spento
java -da -ea:com.mio.AlgoritmoCritico MyApp
\end{lstlisting}

\begin{warning}
Le classi di sistema (come quelle in \texttt{java.util.*}) sono caricate dal \textit{Bootstrap Class Loader} e ignorano i flag \texttt{-ea} e \texttt{-da}. Se sospetti un bug interno al JDK e vuoi attivare le sue asserzioni, devi usare obbligatoriamente \texttt{-esa}.
\end{warning}

\begin{observation}
Sebbene sia possibile attivare le asserzioni da riga di comando, nella pratica quotidiana si utilizzano le impostazioni dell'IDE. E' fondamentale inserire il flag \texttt{-ea} nel campo delle \textbf{VM Options} (parametri per la Virtual Machine) e non tra i \textit{Program Arguments}, altrimenti il comando verrebbe ignorato.
\end{observation}

Le asserzioni non richiedono la ricompilazione del codice per essere attivate o disattivate; questa decisione viene presa dal \textbf{Class Loader} (il sottosistema della JVM che cerca i file \texttt{.class} sul disco, ne verifica l'integrità e li carica nella memoria della JVM.) al momento dell'esecuzione. Quando sono disattivate, il Class Loader ignora tutte le asserzioni, che quindi non pesano sulle prestazioni. 

\begin{warning}[Effetti Collaterali]
La condizione di un'asserzione non deve \textbf{mai} alterare lo stato del programma. Poiché il codice dentro l'\texttt{assert} potrebbe non essere eseguito in produzione, inserire operazioni logiche (come eliminare un file o incrementare un contatore) porterebbe a bug diversi tra l'ambiente di test e quello di produzione.
\end{warning}

\begin{lstlisting}[caption={Esempio di Side-Effect da evitare}]
// SBAGLIATO: l'elemento viene rimosso solo se le asserzioni sono attive
assert lista.remove(oggetto); 

// CORRETTO:
boolean rimosso = lista.remove(oggetto);
assert rimosso : "L'oggetto doveva essere presente nella lista!";
\end{lstlisting}

Se si lavora su sistemi estremamente critici dove anche lo spazio occupato dai file \texttt{.class} e' un problema, si puo' usare una costante \texttt{static final}:

\begin{lstlisting}
public static final boolean DEBUG = true;
// ...
if (DEBUG) assert x >= 0;
\end{lstlisting}

\begin{tip}
Il compilatore Java è intelligente: se \texttt{DEBUG} è \texttt{false}, l'intero blocco \texttt{if} viene rimosso dal bytecode finale perche' considerato "unreachable code".
\end{tip}

\subsection{Quando scegliere le Asserzioni}

Java offre tre strumenti per gestire i fallimenti del sistema: le \textbf{Eccezioni}, il \textbf{Logging} (che vedremo nella prossima sezione) e le \textbf{Asserzioni}. Scegliere quella corretta dipende dalla natura dell'errore.

\begin{itemize}
    \item \textbf{Eccezioni}: Per condizioni recuperabili o errori comunicati all'utente/chiamante.
    \item \textbf{Logging}: Per tenere traccia di eventi e diagnosticare problemi nel tempo.
    \item \textbf{Asserzioni}: Per individuare errori logici interni durante lo sviluppo.
\end{itemize} 

\medskip

Le asserzioni hanno due caratteristiche fondamentali:
\begin{enumerate}
    \item Sono destinate a errori \textbf{fatali e non recuperabili}.
    \item Sono attive \textbf{solo durante lo sviluppo}. C'è una battuta famosa nel settore: "Usare le asserzioni è come indossare un giubbotto di salvataggio finché sei vicino a riva, per poi buttarlo via quando sei in mezzo all'oceano".
\end{enumerate}

\medskip

Il dubbio più comune è: "Devo controllare i parametri di un metodo con un \texttt{if-throw} o con un \texttt{assert}?". La risposta dipende dal \textbf{contratto} del metodo.

Se la documentazione del metodo dichiara esplicitamente che verrà lanciata un'eccezione in caso di parametri errati (usando i tag Javadoc \texttt{@throws}), quel comportamento fa parte del \textbf{contratto}. In questo caso \textbf{non} si devono usare le asserzioni.

\begin{lstlisting}[caption={Esempio di contratto con eccezioni}]
/**
 * @throws IllegalArgumentException se fromIndex > toIndex
 */
static void sort(int[] a, int fromIndex, int toIndex) {
    if (fromIndex > toIndex) 
        throw new IllegalArgumentException("fromIndex > toIndex");
    // ...
}
\end{lstlisting}

Se invece la documentazione pone una \textbf{precondizione} (es. "il parametro $a$ non deve essere \texttt{null"}) senza promettere un'eccezione specifica, il chiamante è avvisato: se viola la precondizione, il comportamento è indefinito. Qui l'asserzione è lo strumento perfetto per il programmatore.

\begin{tip}
In sintesi: se il controllo deve sopravvivere nel prodotto finale perché l'errore può capitare anche se il codice è perfetto (es. input utente), usa le \textbf{Eccezioni}. Se il controllo serve a beccare il programmatore mentre scrive codice sbagliato, usa le \textbf{Asserzioni}.
\end{tip}

\section{Logging}

Tutti i programmatori Java conoscono il rito del "debugging tramite stampa": inserire chiamate a \texttt{System.out.println} nei punti critici del codice per capire cosa stia succedendo. Una volta risolto il problema, queste istruzioni vengono rimosse, per poi essere reinserite al bug successivo.

I framework di \textbf{logging} sono progettati per superare questo ciclo inefficiente, offrendo una soluzione permanente, professionale e configurabile.

Il logging non è una semplice "stampa a video evoluta", ma un sistema di tracciamento che offre:
\begin{itemize}
    \item \textbf{Persistenza}: Registrare i messaggi su file o database, non solo sulla console.
    \item \textbf{Granularità}: Classificare i messaggi per importanza (es. INFO, WARNING, SEVERE).
    \item \textbf{Configurabilità}: Attivare o disattivare i messaggi senza modificare o ricompilare il codice.
\end{itemize}

\medskip

In un'applicazione professionale, l'uso di \texttt{System.out.println} è considerato una cattiva pratica. Considera questo scenario: un server subisce un guasto alle 3 di notte mentre processa dei pagamenti. L' "utente" in questo caso non è un essere umano che interagisce in tempo reale, ma un processo automatico che gira in background. Senza un log persistente su file, l'errore svanirebbe nel nulla e sarebbe impossibile ricostruire l'accaduto il mattino seguente. Il logging è la "scatola nera" per quando non si è presenti a guardare la console.

\subsection{Architettura del Logging: Facciate e Implementazioni}

Nel mondo Java, il logging non è un blocco monolitico. Per garantire flessibilità, si utilizza un'architettura divisa in due parti:

\begin{itemize}
    \item \textbf{Frontend (façade/API)}: È il codice (es. \texttt{logger.log(...)}) che il programmatore scrive. Definisce \textit{come} si inviano i messaggi. Esempi: \textbf{SLF4J}, \textbf{Platform Logging API (JEP 264)}.
    \item \textbf{Backend (Implementation)}: È il motore che decide \textit{dove} vanno i messaggi (file, console, rete) e come formattarli. Esempi: \textbf{java.util.logging (JUL)}, \textbf{Log4j}, \textbf{Logback}.
\end{itemize}

\medskip

Utilizzare una "facciata" (come SLF4J o la Platform API) permette di cambiare il motore di logging in futuro senza dover modificare una singola riga di codice nel resto dell'applicazione. Basta cambiare una configurazione esterna.

Anche se nel mondo enterprise si usano spesso Log4j o Logback, studiare il backend standard del JDK (JUL) è importante per tre motivi:
\begin{enumerate}
    \item \textbf{Fondamenta}: Offre una base solida per capire come funzionano tutti gli altri framework.
    \item \textbf{Semplicità e Sicurezza}: Essendo parte del JDK, è più "snello" e meno soggetto ad attacchi informatici. 
    \item \textbf{Zero Dipendenze}: Non richiede librerie esterne, rendendolo perfetto per piccoli progetti o moduli di sistema.
\end{enumerate} 

\begin{warning}[Sicurezza e Log4j]
La complessità eccessiva può essere un pericolo. Il famoso attacco hacker a \textbf{Log4j} (Log4Shell) è stato possibile proprio a causa di funzionalità oscure e troppo potenti che permettevano l'esecuzione di codice malevolo tramite un semplice messaggio di log.
\end{warning}

Di seguito, salvo avviso contrario, faremo dunque riferimento a JEP 264 per il frontend e a JUL per il backend.

\subsection{L'interfaccia \texttt{System.Logger}}
Come abbiamo accennato alla fine di \ref{sec: classi statiche}, oltre alle classi, si possono annidare anche le interfacce. \texttt{System.Logger} è proprio un esempio di \textbf{Nested Interface}: la classe \textbf{System} funge da \textit{namespace} per l'interfaccia \texttt{Logger} quindi, quando scriviamo \texttt{System.Logger}, il punto (\texttt{.}) non indica l'accesso a un metodo o a una variabile di istanza, ma l'accesso a un \textbf{tipo} definito all'interno dell'ambito (scope) della classe \texttt{System}.

Questa scelta di design non è estetica, ma strutturale:
\begin{itemize}
    \item \textbf{Namespacing}: Evita di creare nomi globali troppo generici come \texttt{Logger} nel pacchetto \texttt{java.lang}, che potrebbero andare in conflitto con altre librerie.
    \item \textbf{Accoppiamento Logico}: Indica chiaramente che l'interfaccia esiste \textit{solo} in funzione dei servizi offerti dalla classe ospite.
    \item \textbf{Incapsulamento}: Permette di raggruppare tipi correlati senza "inquinare" il file system con decine di piccoli file \texttt{.java}.
\end{itemize}

\medskip

Se analizzassimo il codice sorgente del JDK, la struttura apparirebbe qualitativamente così:

\begin{lstlisting}[language=Java, caption={Struttura semplificata di System.java}]
public final class System {
    // Campi e metodi statici (out, err, getProperty...)
    
    /** * Interfaccia annidata statica. 
     * In Java, le interfacce annidate sono implicitamente 'static'.
     */
    public interface Logger {
        enum Level { ALL, TRACE, DEBUG, INFO, WARNING, ERROR, OFF }
        void log(Level level, String msg);
        // ... altri metodi abstract ...
    }

    public static Logger getLogger(String name) {
        // Restituisce un'implementazione concreta fornita dal backend
    }
}
\end{lstlisting}


Ricordiamo che per utilizzare un tipo annidato, la sintassi segue la gerarchia di contenimento:

\begin{enumerate}
    \item \textbf{Dichiarazione del tipo}: si usa il nome della classe esterna come prefisso:\\ \texttt{System.Logger mioLogger;}
    \item \textbf{Accesso ai membri del tipo annidato}: se l'interfaccia annidata contiene a sua volta delle enumerazioni (come \texttt{Level}), la catena si allunga:\\ \texttt{System.Logger.Level.INFO}.
\end{enumerate}


\begin{tip}
Ricorda: un'interfaccia annidata è sempre \texttt{static}, anche se non lo scrivi esplicitamente. Questo significa che può essere utilizzata senza dover creare un'istanza della classe esterna (cosa che nel caso di \texttt{System} sarebbe comunque impossibile, avendo il costruttore privato).
\end{tip}

\subsection{Ottenimento e Utilizzo del Logger}

L'ottenimento di un logger non avviene tramite l'operatore \texttt{new}, ma attraverso il metodo statico \texttt{System.getLogger()}. L'argomento del metodo è una stringa che rappresenta il nome del logger; questo nome può essere arbitrario, ma è considerata una \textit{best practice} utilizzare il nome del package (o della classe) in cui si generano i messaggi. 

Questo approccio permette di sfruttare la \textbf{gerarchia dei nomi} per una configurazione granulare. Di fatto, il sistema utilizza un meccanismo di \textbf{caching} concettualmente analogo a quello della \textit{String Pool}: la prima chiamata al metodo con un determinato nome crea il logger, mentre tutte le chiamate successive con lo stesso nome restituiscono il medesimo oggetto già esistente.

\begin{lstlisting}[language=Java, caption={Recupero di un Logger}]
System.Logger logger = System.getLogger("com.myapp");
\end{lstlisting}

\begin{tip}
Poiché i logger con lo stesso nome sono condivisi, è prassi comune dichiararli come \texttt{private static final}. Questo garantisce che la variabile non venga riassegnata e che sia disponibile a livello di classe.
\end{tip}

Una volta ottenuto il logger, è possibile registrare messaggi specificando il \textbf{livello di importanza}. La sintassi base della \textit{Platform Logging API} è la seguente:

\begin{lstlisting}[language=Java, caption={Esempio di registrazione di un messaggio}]
logger.log(System.Logger.Level.INFO, "Apertura del file " + filename);
\end{lstlisting}

A differenza di una stampa manuale, il sistema di logging produce un record strutturato. Ecco un esempio di come appare il messaggio precedente in console (usando il backend JUL):

\begin{lstlisting}[language=bash]
Apr 28, 2026 10:20:00 AM com.myapp.Main read
INFO: Apertura del file dati.txt
\end{lstlisting}

Il logger si comporta come un segretario scrupoloso. Oltre al messaggio fornito dal programmatore, esso cattura e stampa automaticamente i seguenti \textbf{metadati}:
\begin{itemize}
    \item \textbf{Timestamp}: Data e ora precisa dell'evento.
    \item \textbf{Nome della Classe}: Il nome completo (\textit{Fully Qualified Name}) della classe chiamante, nel nostro caso \texttt{com.myapp.Main}.
    \item \textbf{Nome del Metodo}: Il nome del metodo all'interno del quale è stata invocata la scrittura del log, nel nostro caso \texttt{read}.
    \item \textbf{Livello}: L'etichetta di severità (\texttt{INFO}, \texttt{WARNING}, ecc.).
\end{itemize}

\medskip

Il fatto che il nome della classe e del metodo siano inclusi \textbf{automaticamente} è un vantaggio enorme in fase di debug. Non è più necessario scrivere manualmente \texttt{"Errore nel metodo read(): ..."}, poiché il logger "sa" già dove si trova grazie all'analisi dello stack trace.
