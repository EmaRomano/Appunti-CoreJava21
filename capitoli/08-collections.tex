\section{Separazione tra Interfaccia e Implementazione}

Il \textit{Java Collections Framework} adotta un approccio fondamentale della programmazione a oggetti: la netta separazione tra le \textbf{interfacce} (che definiscono il comportamento) e le \textbf{implementazioni} concrete (che definiscono le strutture dati e le prestazioni). Per illustrare bene questo concetto e i suoi vantaggi, esaminiamo il seguente esempio. 

Una \texttt{Queue} (coda) specifica un comportamento \textbf{FIFO} (\textit{First In, First Out}): aggiunta in coda, rimozione in testa e monitoraggio della dimensione.

\begin{lstlisting}[caption={Interfaccia minimale di una Queue}]
public interface Queue<E> {
    void add(E e);
    E remove();
    int size();
}
\end{lstlisting}

L'interfaccia non dice nulla su come i dati siano memorizzati. Esistono due implementazioni principali nel JDK:
\begin{enumerate}
    \item \textbf{\texttt{ArrayDeque}}: Utilizza un \textit{circular array} (array circolare, vedi in appendice \ref{appendix circ array}).
    \item \textbf{\texttt{LinkedList}}: Utilizza una lista concatenata di nodi.
\end{enumerate}

Il vantaggio principale di questa separazione è la flessibilità. Nel codice backend, è considerata una \textit{best practice} utilizzare il tipo interfaccia per dichiarare le variabili, limitando l'uso della classe concreta solo al momento della costruzione.

\begin{lstlisting}[caption={Utilizzo corretto delle interfacce}]
// Uso l'interfaccia come tipo della variabile
Queue<Customer> expressLane = new ArrayDeque<>(100);
expressLane.add(new Customer("Harry"));

// Se cambio idea, devo cambiare solo la riga del costruttore
Queue<Customer> expressLane = new LinkedList<>();
\end{lstlisting}

\begin{bestpractice}[Coding to an Interface]
Usare il tipo dell'interfaccia (es. \texttt{Queue}) invece del tipo concreto (es. \texttt{ArrayDeque}) protegge il codice da dipendenze eccessive. Se accidentalmente si utilizzasse un metodo specifico di \texttt{ArrayDeque} non presente in \texttt{Queue}, il compilatore segnalerebbe l'errore, garantendo che il codice rimanga facilmente intercambiabile.

In Java 6, quando fu introdotta \texttt{ArrayDeque}, i programmatori che avevano "programmato per interfacce" poterono migliorare istantaneamente le prestazioni delle loro applicazioni cambiando una singola riga di codice (il costruttore), lasciando intatto tutto il resto della logica aziendale.
\end{bestpractice}

Perché scegliere l'una o l'altra? L'interfaccia è agnostica riguardo alle prestazioni, ma le implementazioni no:

\begin{itemize}
    \item \textbf{\texttt{ArrayDeque}}: È generalmente più performante. Ha un impatto minore sul \textit{Garbage Collector} perché non deve creare un oggetto "Nodo" per ogni elemento inserito. Richiede la riallocazione dell'array quando la capacità finisce (operazione computazionalmente costosa), ma se la capacità iniziale è ben calibrata, queste riallocazioni sono rare. 
    \item \textbf{\texttt{LinkedList}}: Non ha problemi di capacità finita, ma ogni elemento aggiunto comporta la creazione di un nuovo oggetto in memoria, rendendola meno efficiente in scenari ad alto volume di dati.
\end{itemize}

\section{L'Interfaccia Collection}

L'interfaccia \texttt{Collection<E>} è la radice della gerarchia per le strutture dati che gestiscono gruppi di oggetti (ad eccezione delle \textit{Map}). Essa definisce un set di operazioni standard che garantiscono l'interoperabilità tra diverse implementazioni. Di seguito, esaminiamo i metodi di uso più comune. 

\subsection*{Operazioni di Query (Ispezione)}

Questi metodi permettono di ottenere informazioni sullo stato della collezione senza modificarla.

\begin{itemize}
    \item \texttt{int size()}: Restituisce il numero di elementi.
    \item \texttt{boolean isEmpty()}: Equivalente a \texttt{size() == 0}.
    \item \texttt{boolean contains(Object o)}: Verifica la presenza di un elemento. 
\end{itemize}


\subsection*{Operazioni di Modifica}

\begin{itemize}
    \item \texttt{boolean add(E element)}: Restituisce \textbf{\texttt{true}} se l'aggiunta ha effettivamente modificato la collezione, \textbf{\texttt{false}} altrimenti. Ad esempio, in una \texttt{List}, il metodo \texttt{add} restituirà quasi sempre \texttt{true}; invece, in un \textbf{\texttt{Set}} (che non ammette duplicati), se provi ad aggiungere un oggetto già presente, il metodo restituirà \texttt{false}, segnalando che l'operazione non ha avuto effetto.
    \item \texttt{boolean remove(Object o)}: Rimuove una singola istanza dell'elemento, se presente. Il valore restituito segue la stessa logica di \texttt{add}
    \item \texttt{void clear()}: Rimuove tutti gli elementi, svuotando la collezione.
\end{itemize}

\subsection*{Operazioni Bulk (di Massa)}

Questi metodi sono fondamentali per manipolare interi set di dati (es. confrontare liste di ID o filtrare risultati). Per \texttt{addAll}, \texttt{removeAll} e \texttt{retainAll} il valore di ritorno indica se la struttura dati è stata mutata, per \texttt{containsAll} il \texttt{boolean} rappresenta l'esito logico del test di inclusione.

\begin{itemize}
    \item \texttt{boolean addAll(Collection<? extends E> c)}: \textbf{Unione}. Aggiunge tutti gli elementi della collezione \texttt{c} alla collezione corrente. 
    \item \texttt{boolean removeAll(Collection<?> c)}: \textbf{Differenza}. Rimuove dalla collezione corrente tutti gli elementi che sono presenti anche in \texttt{c}. 
    \item \texttt{boolean retainAll(Collection<?> c)}: \textbf{Intersezione}. Rimuove dalla collezione corrente tutti gli elementi che \textbf{non} sono contenuti in \texttt{c}, mantenendo solo i comuni. 
    \item \texttt{boolean containsAll(Collection<?> c)}: \textbf{Inclusione}. Verifica se la collezione corrente contiene tutti gli elementi della collezione \texttt{c}. 
\end{itemize}

\subsection*{Conversione in Array}

Nonostante la potenza delle collezioni, a volte è necessario interfacciarsi con API legacy che richiedono array classici.
\begin{itemize}
    \item \texttt{Object[] toArray()}: Restituisce un array di oggetti.
    \item \texttt{<T> T[] toArray(T[] a)}: Versione tipizzata. Se l'array \texttt{a} è abbastanza grande, i dati vengono inseriti lì; altrimenti, viene creato un nuovo array dello stesso tipo.
\end{itemize}

\subsection*{Metodi Default e Approccio Funzionale (Java 8+)}

Con l'introduzione delle lambda, l'interfaccia si è arricchita di metodi \texttt{default} molto potenti:

\begin{itemize}
    \item \texttt{boolean removeIf(Predicate<? super E> filter)}: Rimuove tutti gli elementi che soddisfano una determinata condizione. È molto più efficiente e pulito di un ciclo manuale con iteratore.
    \item \texttt{Stream<E> stream()}: Il punto d'ingresso per la \textit{Stream API}, essenziale per trasformazioni e filtraggi complessi in stile dichiarativo. Tratteremo gli \textit{Stream} in un capitolo successivo. 
\end{itemize}

\subsection{Il Concetto di Operazioni Opzionali}
Quasi tutti i metodi che abbiamo elencato non sono di default e ciò potrebbe indurre a pensare che \textit{tutte} le collezioni concrete in Java implementino tutti questi metodi. Tuttavia, questo può essere fuorviante: il framework Java gestisce la varietà delle strutture dati (es. collezioni a dimensione fissa o non modificabili) attraverso il concetto di \textbf{metodi opzionali}.

Invece di creare decine di sotto-interfacce, i progettisti di Java hanno preferito mantenere l'interfaccia \texttt{Collection} unica. 
Se una specifica implementazione non supporta un metodo (ad esempio il metodo \texttt{add} in una collezione a sola lettura), il metodo viene comunque implementato, ma il suo corpo consiste nel lancio di una eccezione:

\begin{lstlisting}[language=Java]
public boolean add(E e) {
    throw new UnsupportedOperationException();
}
\end{lstlisting}

\begin{warningbox}[Attenzione a Runtime]
Poiché l'obbligo di implementazione è soddisfatto formalmente, il compilatore non segnalerà errori. L'errore emergerà solo a \textbf{runtime}. Questo è un motivo in più per cui è fondamentale conoscere bene la documentazione della classe concreta che si sta istanziando.
\end{warningbox}

\section{Iteratori}

Per scorrere gli elementi di una collezione, il Java Collections Framework non si affida agli indici (che sarebbero inefficienti per strutture come le liste concatenate), ma utilizza l'interfaccia \texttt{Iterator<E>}; essa definisce il meccanismo standard per visitare gli elementi di una collezione uno alla volta. 
\begin{figure}[htbp]
    \centering
    \includegraphics[width=\textwidth]{iterator.pdf} 
    \caption{l'iteratore come cursore \textbf{tra} elementi}
    \label{fig: iteratore}
\end{figure}
A livello astratto, si può immaginare l'iteratore come un cursore posizionato \textbf{tra} gli elementi (si veda la figura \ref{fig: iteratore}).

L'interfaccia \texttt{Iterator} definisce quattro metodi principali:
\begin{lstlisting}[language=Java]
public interface Iterator<E> {
   boolean hasNext();
   E next();
   default void remove();
   default void forEachRemaining(Consumer<? super E> action);
}
\end{lstlisting}

\begin{itemize}
    \item \texttt{E next()}: Chiamando questo metodo, il cursore "salta" oltre l'elemento successivo e restituisce un riferimento all'elemento che ha appena saltato.
    \item \texttt{boolean hasNext()}: Restituisce \texttt{true} se ci sono ancora elementi da scorrere.
    \item \texttt{void remove()}: Cancella l'elemento restituito dall'ultima chamata di \texttt{next}; esso è soggetto a vincoli: può essere chiamato solo dopo una chiamata a \texttt{next} (in concetto è: prima lo devo leggere, poi lo posso cancellare) e solo \textbf{una volta} dopo ogni chiamata a \texttt{next()} (non si può chiamare due volte di fila senza un \texttt{next} in mezzo). Se uno di questi vincoli non viene rispettato, si riceve una \texttt{IllegalStateException}.
    \item \texttt{void forEachRemaining(Consumer<? super E> action)}: Esegue un'azione su tutti gli elementi rimanenti.
\end{itemize}

\begin{dettagli}[]
    Da notare che nel codice dell'interfaccia i metodi \texttt{remove()} e \texttt{forEachRemaining()} sono segnati come default. Questo significa che chi implementa l'interfaccia non è obbligato a scriverne il corpo se non ne ha bisogno.

    Nello specifico, \texttt{remove()} è un metodo "opzionale": alcune collezioni (come quelle non modificabili) potrebbero lanciare una \texttt{UnsupportedOperationException} se provi a usarlo.
\end{dettagli}

Come si vede dalla figura \ref{fig:collection-hierarchy}, \texttt{Iterable} rappresenta l'interfaccia padre di \texttt{Collection}; ogni classe che la implementa dichiara di poter restituire i propri elementi uno alla volta. Essa contiene un solo metodo astratto e non di default che restituisce un iteratore:

\begin{lstlisting}[language=Java]
public interface Iterable<E> {
   Iterator<E> iterator();
}
\end{lstlisting}

Questo metodo viene dunque ereditato da \texttt{Collection}; per ispezionare tutti gli elementi di una collezione, si richiede un iteratore e si richiama ripetutamente il metodo \texttt{next()}. Se si chiama \texttt{next()} quando si è già raggiunta la fine della collezione, l'iteratore solleva una \texttt{NoSuchElementException}. Per questo motivo, è fondamentale verificare sempre la presenza di un elemento successivo tramite il metodo \texttt{hasNext()}.

\begin{lstlisting}[caption={Utilizzo classico del ciclo while con Iterator}]
Collection<String> coll = ...;
Iterator<String> iter = coll.iterator();
while (iter.hasNext()) {
   String element = iter.next();
   // operazione sull'elemento
}
\end{lstlisting}

Java permette di scrivere il ciclo precedente in modo molto più conciso grazie al costrutto \textbf{for each}. Il compilatore traduce automaticamente questo costrutto in un ciclo con iteratore.

\begin{lstlisting}[caption={Sintassi for each}]
for (String element : coll) {
   // operazione sull'elemento
}
\end{lstlisting}

\begin{dettagli}
    \begin{lstlisting}[language=Java]
        // Scritto dallo sviluppatore:
        for (String s : coll) { System.out.println(s); }

        // Tradotto dal compilatore:
            Iterator<String> it = coll.iterator();
            while (it.hasNext()) {
                String s = it.next();
                System.out.println(s);
        }
    \end{lstlisting}
\end{dettagli}

Il ciclo "for each" può essere utilizzato con qualsiasi oggetto che implementi l'interfaccia \textbf{\texttt{Iterable}}. 

\begin{bestpractice}[]
Poiché l'interfaccia \texttt{Collection} estende \texttt{Iterable}, ogni collezione del framework Java può essere scorsa usando la sintassi \textit{for each}. Prediligi sempre questa sintassi per la leggibilità, a meno che tu non abbia bisogno di rimuovere elementi durante l'iterazione.
\end{bestpractice}

\begin{warningbox}[Effetti Collaterali]
Tieni presente che durante un \texttt{forEach}, non puoi modificare la struttura della collezione (es. aggiungere o rimuovere elementi), altrimenti Java lancerà una \texttt{ConcurrentModificationException}.
\end{warningbox}

Oltre ai cicli tradizionali, il \textit{Collections Framework} mette a disposizione metodi che accettano \textbf{lambda expressions} per elaborare gli elementi in modo più conciso e leggibile. Invece di gestire manualmente l'iteratore, possiamo passare un'azione (sotto forma di lambda) direttamente alla collezione o all'iteratore stesso.

\begin{lstlisting}[caption={Esempi di iterazione funzionale}]
// Applica l'azione a ogni elemento della collezione
myCollection.forEach(element -> System.out.println(element));

// Applica l'azione agli elementi rimanenti dell'iteratore
myIterator.forEachRemaining(element -> System.out.println(element));
\end{lstlisting}

Entrambi questi metodi utilizzano l'interfaccia funzionale \texttt{Consumer<? super E>}. La lambda viene invocata internamente per ogni elemento della collezione senza che lo sviluppatore debba preoccuparsi di chiamare \texttt{next()} o \texttt{hasNext()}.

Per quanto riguarda l'ordine in cui gli elementi vengono visitati, il comportamento dipende esclusivamente dal tipo di collezione; ad esempio, per un {\texttt{ArrayList}} l'iteratore è \textbf{predicibile}: inizia dall'indice 0 e incrementa di uno a ogni passo; per un \textbf{\texttt{HashSet}}, l'iteratore visita gli elementi in ordine \textbf{essenzialmente casuale} e non si può fare alcuna assunzione sulla sequenza di uscita.

\begin{tip}[Scelta della Collezione]
Se l'ordine dei dati è irrilevante (es. sommare dei totali o contare occorrenze), l'incertezza sull'ordine di visita non è un problema. Se invece l'ordine è parte della logica di business (es. una lista di transazioni cronologiche), è vitale scegliere una collezione che garantisca l'ordinamento come \texttt{ArrayList} o \texttt{TreeSet}.
\end{tip}
