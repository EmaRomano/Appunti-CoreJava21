\section{Il Class Path}

Il \textbf{Class Path} (percorso delle classi) è la collezione di tutte le locazioni (cartelle o archivi) in cui la JVM e il compilatore cercano i file \texttt{.class}. È, di fatto, l'elenco dei punti di partenza per la risoluzione dei package.

Il Class Path è una stringa composta da vari percorsi. Il separatore dipende dal sistema operativo:

\begin{table}[h]
    \centering
    \small % Riduce leggermente la dimensione del testo per far stare tutto comodamente
    \begin{tabularx}{\textwidth}{l X l} % Rimosse le linee verticali per stile professionale
        \toprule
        \textbf{Sistema} & \textbf{Esempio di Class Path} & \textbf{Separatore} \\
        \midrule
        \textbf{Fedora / UNIX} & \path{/home/user/classdir:.:/libs/archive.jar} & Due punti (\texttt{:}) \\
        \addlinespace % Aggiunge un piccolo spazio tra le righe
        \textbf{Windows} & \path{C:\classdir;.;C:\libs\archive.jar} & Punto e virgola (\texttt{;}) \\
        \bottomrule
    \end{tabularx}
\end{table}

\begin{observation}[Il simbolo del punto]
    In entrambi i sistemi, il punto (\texttt{.}) rappresenta la \textbf{directory corrente}.
\end{observation}

È comune confondere il Classpath con la directory di output. Ecco la distinzione corretta:

\begin{itemize}
    \item \textbf{Per il Compilatore (\texttt{javac})}: Il Classpath serve a \textbf{risolvere i riferimenti esterni}. Se la tua classe \texttt{A} usa la classe \texttt{B} (che si trova in un JAR o in un'altra cartella), il compilatore deve "leggere" \texttt{B} per verificare che tu stia usando i suoi metodi correttamente.
    \begin{itemize}
        \item \textit{Nota}: Per decidere \textbf{dove mettere} il file compilato, si usa sempre il flag \texttt{-d}, non il Classpath.
    \end{itemize}

    \item \textbf{Per la JVM (\texttt{java})}: Il Classpath serve a \textbf{caricare il codice in memoria}. La JVM non sa nulla dei file \texttt{.java}; lei cerca solo il bytecode (\texttt{.class}) seguendo i percorsi indicati nel Classpath per poter avviare l'esecuzione.
\end{itemize}

\subsection{La "Ricerca Intelligente" del Compilatore}
Horstmann sottolinea che il compilatore è più sofisticato della JVM nella ricerca. Quando incontra una classe, ad esempio \texttt{Employee}:
\begin{enumerate}
    \item Cerca il file \texttt{Employee.class} nel Classpath.
    \item Se trova anche il file \texttt{Employee.java}, controlla le date di modifica.
    \item \textbf{Auto-recompile}: Se il sorgente (\texttt{.java}) è più recente del compilato (\texttt{.class}), il compilatore ricompila automaticamente il sorgente per garantire che tu stia usando la versione più aggiornata.
\end{enumerate}

\subsection{I file JAR (Java Archive)}
Oltre alle normali directory, le classi possono essere raggruppate in file \textbf{JAR}.
\begin{itemize}
    \item Un file JAR è essenzialmente un archivio in formato \textbf{ZIP} che contiene classi e strutture di sottocartelle compresse.
    \item È lo standard per la distribuzione di librerie di terze parti.
    \item Permette di migliorare le prestazioni (un solo file da aprire invece di centinaia) e risparmiare spazio.
\end{itemize}

\subsection{L'Algoritmo di Ricerca}
Se la JVM deve caricare la classe \texttt{com.horstmann.corejava.Employee}, cercherà in ordine:
\begin{enumerate}
    \item Nelle classi delle \textbf{API di Java} (sempre cercate per prime, non serve includerle nel Class Path).
    \item In ogni voce del Class Path, aggiungendo il percorso del package alla base. Ad esempio, se il path contiene \texttt{/home/user/classdir}, cercherà: \\
    \texttt{/home/user/classdir/com/horstmann/corejava/Employee.class}
\end{enumerate}

\begin{caution}[La trappola del punto]
    Per impostazione predefinita, il Class Path è costituito solo dal punto (\texttt{.}). Tuttavia, se specifichi un Class Path personalizzato (tramite la variabile d'ambiente \texttt{CLASSPATH} o il flag \texttt{-cp}) e \textbf{dimentichi} di includere il punto (\texttt{.}), la JVM non guarderà più nella cartella corrente.
    \begin{itemize}
        \item \textbf{Risultato}: Il programma compila (perché \texttt{javac} guarda sempre la cartella corrente), ma non parte (perché \texttt{java} non la trova).
    \end{itemize}
\end{caution}

\subsection{Uso delle Wildcard}
È possibile includere tutti i file JAR di una cartella usando l'asterisco:
\begin{lstlisting}[language=bash]
# Su Fedora (bisogna proteggere l'asterisco dalle espansioni della shell)
java -cp "/home/user/libs/*:." MiaClasse
\end{lstlisting}
\textbf{Nota}: L'asterisco include tutti i file \texttt{.jar}, ma non i singoli file \texttt{.class} sparsi nella cartella.

\section{La Classe \texttt{java.lang.System}}

La classe \texttt{java.lang.System} rappresenta uno dei pilastri fondamentali del Java Development Kit (JDK). Situata nel pacchetto \texttt{java.lang}, essa funge da interfaccia primaria tra l'applicazione Java e l'ambiente in cui viene eseguita (Java Virtual Machine e Sistema Operativo).

\subsection{Caratteristiche Strutturali}

Dal punto di vista dell'architettura software, la classe \texttt{System} presenta caratteristiche peculiari:
\begin{itemize}
    \item \textbf{Finalità}: È dichiarata come \texttt{public final}, il che impedisce qualsiasi tentativo di ereditarietà.
    \item \textbf{Istanziabilità}: Il costruttore è privato, rendendo impossibile creare istanze della classe. Tutte le sue funzionalità sono accessibili esclusivamente tramite metodi e campi \texttt{static}.
    \item \textbf{Accessibilità}: Essendo parte di \texttt{java.lang}, viene importata automaticamente in ogni file sorgente Java.
\end{itemize}

\subsection{I Flussi di I/O Standard}

La classe gestisce i tre canali di comunicazione standard definiti a livello di sistema operativo. Questi campi sono istanze di \texttt{PrintStream} (per l'output) e \texttt{InputStream} (per l'input).

\begin{enumerate}
    \item \texttt{System.out}: Rappresenta lo \textit{standard output}. Viene utilizzato per l'invio di dati alla console o al terminale.
    \item \texttt{System.err}: Rappresenta lo \textit{standard error}. Sebbene tecnicamente simile a \texttt{out}, è destinato specificamente alla diagnostica. Molti IDE e sistemi di logging separano questi due flussi, evidenziando spesso l'errore in rosso.
    \item \texttt{System.in}: Rappresenta lo \textit{standard input}. Viene solitamente utilizzato in combinazione con la classe \texttt{Scanner} per leggere l'input dell'utente dalla tastiera.
\end{enumerate}

\begin{lstlisting}[language=Java, caption={Esempio di utilizzo dei flussi standard}]
System.out.println("Messaggio informativo");
System.err.println("Messaggio di errore critico");
\end{lstlisting}

\subsection{Gestione del Tempo e Misurazioni}

Per il monitoraggio delle prestazioni e la gestione temporale, la classe offre due metodi fondamentali con risoluzioni differenti:

\begin{itemize}
    \item \texttt{currentTimeMillis()}: Restituisce il tempo corrente in millisecondi dall'epoca Unix (1 gennaio 1970). È ideale per calcolare date o timestamp da salvare nel database.
    \item \texttt{nanoTime()}: Restituisce il valore di un timer ad alta risoluzione in nanosecondi. Non è legato all'ora solare, ma è lo strumento standard per misurare quanto tempo impiega un blocco di codice ad essere eseguito (benchmarking).
\end{itemize}

\subsection{Interazione con l'Ambiente Operativo}

La classe permette di interrogare il sistema operativo e la configurazione della JVM tramite le \textit{System Properties} e le \textit{Environment Variables}.

\subsubsection{Proprietà di Sistema}
Le proprietà di sistema sono coppie chiave-valore che definiscono il runtime Java.
\begin{itemize}
    \item \texttt{System.getProperty("user.home")}: Restituisce il percorso della cartella utente.
    \item \texttt{System.getProperty("java.version")}: Restituisce la versione di Java in uso.
    \item \texttt{System.lineSeparator()}: Restituisce il separatore di riga specifico dell'OS (\texttt{\textbackslash n} su Unix, \texttt{\textbackslash r\textbackslash n} su Windows).
\end{itemize}

\subsubsection{Variabili d'Ambiente}
Le variabili d'ambiente sono impostate a livello di OS e sono fondamentali nel backend moderno (specialmente con Docker e Kubernetes) per gestire configurazioni sensibili come password o indirizzi di database.
\begin{lstlisting}[language=Java]
String dbUrl = System.getenv("DATABASE_URL");
\end{lstlisting}

\subsection{Utility di Sistema e Memoria}

\begin{itemize}
    \item \textbf{\texttt{arraycopy()}}: Un metodo nativo estremamente performante per copiare dati tra array. A differenza di un ciclo \texttt{for}, questo metodo sposta i blocchi di memoria direttamente tramite chiamate a basso livello.
    \item \textbf{\texttt{gc()}}: Suggerisce alla JVM di eseguire il Garbage Collector. È importante notare che è solo un suggerimento e non garantisce l'immediata liberazione della memoria.
    \item \textbf{\texttt{exit(int status)}}: Termina la JVM corrente. Un codice di stato \texttt{0} indica una chiusura corretta, mentre un valore diverso da zero indica un fallimento.
\end{itemize}

\subsection{Integrazione con il Logging Moderno}

Dalla versione 9 di Java, la classe \texttt{System} ha assunto un ruolo centrale nella \textit{Platform Logging API} tramite il metodo:
\begin{lstlisting}[language=Java]
System.Logger logger = System.getLogger("NomeLogger");
\end{lstlisting}
Questo permette di astrarre il sistema di logging, delegando a \texttt{System} il compito di trovare e utilizzare il miglior backend disponibile (come JUL o Log4j).

\begin{warningbox}
Nel contesto dei server e delle applicazioni web, l'uso di \texttt{System.exit()} e la manipolazione diretta di \texttt{System.out} sono sconsigliati. Si preferisce delegare la chiusura al contenitore dell'applicazione e utilizzare i framework di logging per l'output, garantendo così la stabilità e la tracciabilità del sistema.
\end{warningbox}

\section{Logging [BOZZE DA RIPRENDERE]}

Tutti i programmatori Java conoscono il rito del "debugging tramite stampa": inserire chiamate a \texttt{System.out.println} nei punti critici del codice per capire cosa stia succedendo. Una volta risolto il problema, queste istruzioni vengono rimosse, per poi essere reinserite al bug successivo.

I framework di \textbf{logging} sono progettati per superare questo ciclo inefficiente, offrendo una soluzione permanente, professionale e configurabile.

Il logging non è una semplice "stampa a video evoluta", ma un sistema di tracciamento che offre:
\begin{itemize}
    \item \textbf{Persistenza}: Registrare i messaggi su file o database, non solo sulla console.
    \item \textbf{Granularità}: Classificare i messaggi per importanza (es. INFO, WARNING, SEVERE).
    \item \textbf{Configurabilità}: Attivare o disattivare i messaggi senza modificare o ricompilare il codice.
\end{itemize}

\medskip

In un'applicazione professionale, l'uso di \texttt{System.out.println} è considerato una cattiva pratica. Considera questo scenario: un server subisce un guasto alle 3 di notte mentre processa dei pagamenti. L' "utente" in questo caso non è un essere umano che interagisce in tempo reale, ma un processo automatico che gira in background. Senza un log persistente su file, l'errore svanirebbe nel nulla e sarebbe impossibile ricostruire l'accaduto il mattino seguente. Il logging è la "scatola nera" per quando non si è presenti a guardare la console.

\subsection{Architettura del Logging: Facciate e Implementazioni}

Nel mondo Java, il logging non è un blocco monolitico. Per garantire flessibilità, si utilizza un'architettura divisa in due parti:

\begin{itemize}
    \item \textbf{Frontend (façade/API)}: È il codice (es. \texttt{logger.log(...)}) che il programmatore scrive. Definisce \textit{come} si inviano i messaggi. Esempi: \textbf{SLF4J}, \textbf{Platform Logging API (JEP 264)}.
    \item \textbf{Backend (Implementation)}: È il motore che decide \textit{dove} vanno i messaggi (file, console, rete) e come formattarli. Esempi: \textbf{java.util.logging (JUL)}, \textbf{Log4j}, \textbf{Logback}.
\end{itemize}

\medskip

Utilizzare una "facciata" (come SLF4J o la Platform API) permette di cambiare il motore di logging in futuro senza dover modificare una singola riga di codice nel resto dell'applicazione. Basta cambiare una configurazione esterna.

Anche se nel mondo enterprise si usano spesso Log4j o Logback, studiare il backend standard del JDK (JUL) è importante per tre motivi:
\begin{enumerate}
    \item \textbf{Fondamenta}: Offre una base solida per capire come funzionano tutti gli altri framework.
    \item \textbf{Semplicità e Sicurezza}: Essendo parte del JDK, è più "snello" e meno soggetto ad attacchi informatici. 
    \item \textbf{Zero Dipendenze}: Non richiede librerie esterne, rendendolo perfetto per piccoli progetti o moduli di sistema.
\end{enumerate} 

\begin{warningbox}[Sicurezza e Log4j]
La complessità eccessiva può essere un pericolo. Il famoso attacco hacker a \textbf{Log4j} (Log4Shell) è stato possibile proprio a causa di funzionalità oscure e troppo potenti che permettevano l'esecuzione di codice malevolo tramite un semplice messaggio di log.
\end{warningbox}

Di seguito, salvo avviso contrario, faremo dunque riferimento a JEP 264 per il frontend e a JUL per il backend.

\subsection{L'interfaccia \texttt{System.Logger}}
Come abbiamo accennato alla fine di \ref{sec: classi statiche}, oltre alle classi, si possono annidare anche le interfacce. \texttt{System.Logger} è proprio un esempio di \textbf{Nested Interface}: la classe \textbf{System} funge da \textit{namespace} per l'interfaccia \texttt{Logger} quindi, quando scriviamo \texttt{System.Logger}, il punto (\texttt{.}) non indica l'accesso a un metodo o a una variabile di istanza, ma l'accesso a un \textbf{tipo} definito all'interno dell'ambito (scope) della classe \texttt{System}.

Questa scelta di design non è estetica, ma strutturale:
\begin{itemize}
    \item \textbf{Namespacing}: Evita di creare nomi globali troppo generici come \texttt{Logger} nel pacchetto \texttt{java.lang}, che potrebbero andare in conflitto con altre librerie.
    \item \textbf{Accoppiamento Logico}: Indica chiaramente che l'interfaccia esiste \textit{solo} in funzione dei servizi offerti dalla classe ospite.
    \item \textbf{Incapsulamento}: Permette di raggruppare tipi correlati senza "inquinare" il file system con decine di piccoli file \texttt{.java}.
\end{itemize}

\medskip

Se analizzassimo il codice sorgente del JDK, la struttura apparirebbe qualitativamente così:

\begin{lstlisting}[language=Java, caption={Struttura semplificata di System.java}]
public final class System {
    // Campi e metodi statici (out, err, getProperty...)
    
    /** * Interfaccia annidata statica. 
     * In Java, le interfacce annidate sono implicitamente 'static'.
     */
    public interface Logger {
        enum Level { ALL, TRACE, DEBUG, INFO, WARNING, ERROR, OFF }
        void log(Level level, String msg);
        // ... altri metodi abstract ...
    }

    public static Logger getLogger(String name) {
        // Restituisce un'implementazione concreta fornita dal backend
    }
}
\end{lstlisting}


Ricordiamo che per utilizzare un tipo annidato, la sintassi segue la gerarchia di contenimento:

\begin{enumerate}
    \item \textbf{Dichiarazione del tipo}: si usa il nome della classe esterna come prefisso:\\ \texttt{System.Logger mioLogger;}
    \item \textbf{Accesso ai membri del tipo annidato}: se l'interfaccia annidata contiene a sua volta delle enumerazioni (come \texttt{Level}), la catena si allunga:\\ \texttt{System.Logger.Level.INFO}.
\end{enumerate}


\begin{tip}
Ricorda: un'interfaccia annidata è sempre \texttt{static}, anche se non lo scrivi esplicitamente. Questo significa che può essere utilizzata senza dover creare un'istanza della classe esterna (cosa che nel caso di \texttt{System} sarebbe comunque impossibile, avendo il costruttore privato).
\end{tip}

\subsection{Ottenimento e Utilizzo del Logger}

L'ottenimento di un logger non avviene tramite l'operatore \texttt{new}, ma attraverso il metodo statico \texttt{System.getLogger()}. L'argomento del metodo è una stringa che rappresenta il nome del logger; questo nome può essere arbitrario, ma è considerata una \textit{best practice} utilizzare il nome del package (o della classe) in cui si generano i messaggi. 

Questo approccio permette di sfruttare la \textbf{gerarchia dei nomi} per una configurazione granulare. Di fatto, il sistema utilizza un meccanismo di \textbf{caching} concettualmente analogo a quello della \textit{String Pool}: la prima chiamata al metodo con un determinato nome crea il logger, mentre tutte le chiamate successive con lo stesso nome restituiscono il medesimo oggetto già esistente.

\begin{lstlisting}[language=Java, caption={Recupero di un Logger}]
System.Logger logger = System.getLogger("com.myapp");
\end{lstlisting}

\begin{tip}
Poiché i logger con lo stesso nome sono condivisi, è prassi comune dichiararli come \texttt{private static final}. Questo garantisce che la variabile non venga riassegnata e che sia disponibile a livello di classe.
\end{tip}

Una volta ottenuto il logger, è possibile registrare messaggi specificando il \textbf{livello di importanza}. La sintassi base della \textit{Platform Logging API} è la seguente:

\begin{lstlisting}[language=Java, caption={Esempio di registrazione di un messaggio}]
logger.log(System.Logger.Level.INFO, "Apertura del file " + filename);
\end{lstlisting}

A differenza di una stampa manuale, il sistema di logging produce un record strutturato. Ecco un esempio di come appare il messaggio precedente in console (usando il backend JUL):

\begin{lstlisting}[language=bash]
Apr 28, 2026 10:20:00 AM com.myapp.Main read
INFO: Apertura del file dati.txt
\end{lstlisting}

Il logger si comporta come un segretario scrupoloso. Oltre al messaggio fornito dal programmatore, esso cattura e stampa automaticamente i seguenti \textbf{metadati}:
\begin{itemize}
    \item \textbf{Timestamp}: Data e ora precisa dell'evento.
    \item \textbf{Nome della Classe}: Il nome completo (\textit{Fully Qualified Name}) della classe chiamante, nel nostro caso \texttt{com.myapp.Main}.
    \item \textbf{Nome del Metodo}: Il nome del metodo all'interno del quale è stata invocata la scrittura del log, nel nostro caso \texttt{read}.
    \item \textbf{Livello}: L'etichetta di severità (\texttt{INFO}, \texttt{WARNING}, ecc.).
\end{itemize}

\medskip

Il fatto che il nome della classe e del metodo siano inclusi \textbf{automaticamente} è un vantaggio enorme in fase di debug. Non è più necessario scrivere manualmente \texttt{"Errore nel metodo read(): ..."}, poiché il logger "sa" già dove si trova grazie all'analisi dello stack trace.
